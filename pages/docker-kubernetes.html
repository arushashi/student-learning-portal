Docker & Kubernetes

Understand containerization with Docker and container orchestration with Kubernetes — the foundational infrastructure technologies that enable scalable, portable, and resilient microservice deployments in modern cloud-native architectures.

Definition

What are Docker & Kubernetes?

Docker is a platform for building, shipping, and running applications inside lightweight, portable containers. A container packages an application with all its dependencies — code, runtime, system libraries, and configuration — into a single, isolated unit that runs consistently across any environment. Unlike traditional virtual machines, containers share the host operating system's kernel, making them significantly lighter (megabytes vs. gigabytes) and faster to start (seconds vs. minutes).

Kubernetes (often abbreviated K8s) is an open-source container orchestration platform originally developed by Google, now maintained by the Cloud Native Computing Foundation (CNCF). It automates the deployment, scaling, and management of containerized applications across clusters of machines. Kubernetes handles service discovery, load balancing, rolling updates, self-healing (restarting failed containers), secret management, and horizontal auto-scaling. Together, Docker provides the container runtime and image format, while Kubernetes manages how and where those containers run at scale.

Key Insight: Docker solved the "it works on my machine" problem by packaging applications with their entire dependency tree into portable containers. Kubernetes solved the "how do I run thousands of these containers reliably" problem by providing a declarative infrastructure layer that continuously reconciles the actual state of your cluster with your desired state. You declare "I want 5 replicas of my API service with 512MB RAM each" and Kubernetes ensures exactly that, automatically replacing crashed containers, distributing load, and scaling up during traffic spikes.

Detailed Explanation

Understanding the Concept

Think of Docker containers like standardized shipping containers in global trade. Before shipping containers, goods had to be manually loaded and unloaded at every port — different shapes, sizes, and handling requirements made logistics a nightmare. Standardized shipping containers revolutionized trade by providing a uniform format that works with any ship, truck, or crane. Docker does the same for software: your application is packaged into a standard container format that runs identically on a developer's laptop, a CI server, and a production cloud cluster. Kubernetes, then, is like the global port management system — it decides which containers go on which ships (servers), reroutes containers when a ship breaks down, and scales the fleet up or down based on demand.

  • Docker Images and Layers: A Docker image is built from a Dockerfile — a set of instructions that define the application's environment. Each instruction (FROM, COPY, RUN, etc.) creates a read-only layer. Layers are cached and shared between images, so if ten microservices all use the same base Node.js image, that base layer is stored only once. When a container runs, Docker adds a thin read-write layer on top of the image layers. Images are stored in registries (Docker Hub, Amazon ECR, Google Container Registry) and pulled by the runtime when deploying. Multi-stage builds allow you to use a heavy build environment (compilers, dev tools) in the first stage and copy only the compiled artifact to a minimal production image, dramatically reducing image size.
  • Kubernetes Architecture — Pods, Services, Deployments: The smallest deployable unit in Kubernetes is a Pod — one or more tightly coupled containers that share networking and storage. A Deployment manages a set of identical Pods, handling rolling updates (gradually replacing old Pods with new ones) and rollbacks. A Service provides a stable network endpoint (IP and DNS name) for a set of Pods, acting as an internal load balancer. Ingress resources manage external HTTP traffic routing. The control plane (API server, scheduler, controller manager, etcd) manages cluster state, while worker nodes run the actual container workloads via kubelet and a container runtime (containerd, CRI-O).
  • Scaling and Self-Healing: Kubernetes supports both horizontal and vertical scaling. Horizontal Pod Autoscaler (HPA) adjusts the number of Pod replicas based on CPU utilization, memory usage, or custom metrics (e.g., queue depth, request latency). Vertical Pod Autoscaler (VPA) adjusts CPU and memory resource requests. Self-healing is built into Kubernetes' reconciliation loop: if a Pod crashes, the ReplicaSet controller detects the discrepancy between desired and actual state and immediately schedules a replacement. Liveness probes detect hung processes, and readiness probes ensure traffic is only sent to Pods that are ready to serve requests.
  • Networking, Storage, and Configuration: Kubernetes networking follows a flat network model where every Pod gets its own IP address and can communicate with any other Pod without NAT. Network Policies provide firewall-like rules between Pods. Persistent Volumes (PVs) and Persistent Volume Claims (PVCs) abstract storage provisioning for stateful workloads (databases, file stores). ConfigMaps store non-sensitive configuration, while Secrets store sensitive data like API keys and database credentials. Together, these abstractions let you define your entire application infrastructure as declarative YAML manifests, enabling GitOps workflows where infrastructure changes go through the same code review and CI/CD process as application code.

Real-World Example

Spotify: Migrating to Kubernetes for 1000+ Services

Spotify migrated over 1,000 microservices from their custom-built orchestration system (Helios) to Kubernetes to standardize their infrastructure and improve developer productivity. Before Kubernetes, each team managed deployment configurations differently, creating operational inconsistency across the organization. With Kubernetes, Spotify established a uniform deployment model: every service is a Docker container defined by a Dockerfile, deployed as a Kubernetes Deployment with associated Services for internal communication. Their platform team built an internal developer portal (Backstage, which they later open-sourced) on top of Kubernetes that abstracts away infrastructure complexity. Engineers deploy by pushing code — CI/CD pipelines build Docker images, push them to their container registry, and update Kubernetes Deployments automatically. Kubernetes' self-healing capabilities mean that when containers crash or nodes fail, services automatically recover without manual intervention. Auto-scaling handles their daily traffic patterns, scaling music streaming services up during morning commutes and down during off-hours, optimizing their cloud infrastructure costs while maintaining reliability for 500+ million users worldwide.

Key Points

Containers vs VMs

Virtual machines emulate entire hardware stacks — each VM runs its own full operating system on top of a hypervisor (VMware, KVM, Hyper-V), consuming gigabytes of RAM and taking minutes to boot. Containers share the host OS kernel and use Linux namespaces and cgroups for isolation, making them lightweight (megabytes), fast to start (sub-second), and efficient in resource usage. A single server can run hundreds of containers versus tens of VMs. However, containers provide weaker isolation than VMs since they share a kernel — a kernel vulnerability could potentially affect all containers. For this reason, some environments use lightweight VMs (Firecracker, gVisor, Kata Containers) that combine VM-level isolation with near-container performance for multi-tenant or security-sensitive workloads.

Docker Images & Layers

Docker images are built from Dockerfiles using a layered filesystem. Each instruction creates a new immutable layer: FROM node:18-alpine establishes the base layer, COPY package.json . creates another, and RUN npm install creates a third. Layers are cached — if you change your application code but not dependencies, only the application layer rebuilds while the npm install layer is reused from cache. Multi-stage builds let you separate build-time dependencies from runtime: compile in a full SDK image, then copy the binary to a minimal base image (Alpine Linux, Distroless) for production. Best practices include ordering Dockerfile instructions from least to most frequently changed, using .dockerignore to exclude unnecessary files, running as non-root users, and scanning images for vulnerabilities with tools like Trivy or Snyk.

K8s Pods, Services & Deployments

A Pod is the atomic deployment unit in Kubernetes — one or more containers sharing a network namespace (same IP, shared localhost) and storage volumes. Deployments manage Pod lifecycle declaratively: you specify desired replicas, container image, resource limits, and update strategy. Rolling updates gradually replace old Pods with new ones, maintaining availability — you can configure maxSurge (extra Pods during update) and maxUnavailable (how many can be down). Services provide stable endpoints: ClusterIP for internal access, NodePort for external access on a static port, and LoadBalancer for cloud provider load balancers. Ingress controllers (NGINX, Traefik) manage HTTP routing, TLS termination, and path-based routing for external traffic.

Self-Healing & Auto-Scaling

Kubernetes continuously reconciles desired state with actual state through its control loop. If a Pod crashes, the ReplicaSet controller immediately schedules a replacement on a healthy node. Liveness probes (HTTP checks, TCP connections, or command execution) detect unresponsive processes and trigger container restarts. Readiness probes remove unhealthy Pods from Service endpoints so they stop receiving traffic. The Horizontal Pod Autoscaler (HPA) adjusts replica counts based on CPU, memory, or custom metrics — for example, scaling a queue worker service based on the number of pending messages. The Cluster Autoscaler adds or removes worker nodes from the cluster based on pending Pod resource requests, ensuring capacity matches demand while minimizing cloud costs.

Quick Summary

  • Docker containers package applications with all dependencies into portable, lightweight units that run consistently across environments
  • Container images use a layered filesystem with caching, making builds efficient and image sharing storage-effective
  • Kubernetes orchestrates containers across clusters, handling deployment, scaling, networking, and self-healing automatically
  • Pods are the smallest K8s unit; Deployments manage replica sets; Services provide stable networking endpoints
  • HPA scales Pods horizontally based on metrics; Cluster Autoscaler scales the underlying node infrastructure
  • Declarative YAML manifests enable GitOps workflows where infrastructure is versioned and reviewed like application code

Interview Questions

Q1: What are the key differences between containers and virtual machines?

Containers and VMs both provide isolation but at fundamentally different levels of the stack:

  • Architecture: VMs run on a hypervisor and each includes a full guest operating system (kernel, system libraries, init system). Containers share the host OS kernel and use Linux namespaces (PID, network, mount, user) and cgroups for isolation. This means containers are much lighter — a container image might be 50MB while a VM image is 5GB.
  • Performance: Containers start in milliseconds to seconds since they don't need to boot an OS. VMs take minutes to boot. Containers have near-native performance since there's no hypervisor overhead. VMs incur virtualization overhead for CPU, memory, and I/O operations.
  • Isolation: VMs provide stronger isolation because each has its own kernel — a vulnerability in one VM's kernel doesn't affect others. Containers share a kernel, so kernel vulnerabilities could potentially affect all containers on a host. This is why security-sensitive environments sometimes use lightweight VMs (Firecracker) or sandboxed runtimes (gVisor).
  • Density: A single host can run hundreds of containers but typically only tens of VMs. This makes containers more cost-efficient for microservice architectures where you're running many small services.

Q2: How does Kubernetes handle rolling updates and rollbacks?

Kubernetes Deployments manage rolling updates through a careful, configurable process that maintains availability:

  • Rolling Update Strategy: When you update a Deployment's container image, Kubernetes creates a new ReplicaSet with the updated Pod template. It gradually scales up the new ReplicaSet while scaling down the old one. Two parameters control the pace: maxSurge (how many extra Pods can exist during the update, e.g., 25%) and maxUnavailable (how many Pods can be unavailable, e.g., 25%).
  • Readiness Gates: Before a new Pod receives traffic, it must pass its readiness probe. This prevents routing requests to Pods that are still starting up (loading caches, warming connections). If new Pods fail readiness checks, the rollout stalls, preventing a bad deployment from progressing.
  • Rollback: Kubernetes stores revision history for Deployments. If a new version has issues, you can rollback with kubectl rollout undo deployment/my-app, which reverts to the previous ReplicaSet's Pod template. You can also rollback to a specific revision using --to-revision=N.
  • Pause/Resume: You can pause a rollout mid-way with kubectl rollout pause to perform canary-style verification on the partial deployment, then resume or rollback based on observations.

Q3: Explain how Kubernetes networking works. How do Pods communicate?

Kubernetes implements a flat networking model with several layers of abstraction:

  • Pod-to-Pod: Every Pod gets a unique IP address within the cluster. Kubernetes requires that all Pods can communicate with every other Pod directly using these IPs, without NAT. This is implemented by Container Network Interface (CNI) plugins like Calico, Flannel, Cilium, or Weave. Containers within the same Pod share a network namespace and communicate via localhost.
  • Services: Since Pod IPs are ephemeral (Pods are recreated frequently), Services provide stable endpoints. A ClusterIP Service gets a virtual IP and DNS name (e.g., my-service.default.svc.cluster.local). kube-proxy programs iptables or IPVS rules on each node to load-balance traffic across the Service's backing Pods based on label selectors.
  • External Access: NodePort Services expose a port on every cluster node. LoadBalancer Services provision cloud provider load balancers. Ingress resources define HTTP routing rules (host-based, path-based) and TLS termination, implemented by Ingress controllers like NGINX or Traefik.
  • Network Policies: By default, all Pods can communicate freely. Network Policies act as Pod-level firewalls, specifying which Pods can communicate on which ports. For example, you can restrict database Pods to only accept connections from application Pods in the same namespace.

Q4: How would you run a stateful application (like a database) on Kubernetes?

Stateful applications require stable identities and persistent storage, which Kubernetes handles through StatefulSets and storage abstractions:

  • StatefulSets: Unlike Deployments where Pods are interchangeable, StatefulSets provide each Pod with a stable hostname (e.g., mysql-0, mysql-1, mysql-2), ordered startup and shutdown (mysql-0 starts before mysql-1), and persistent storage that follows the Pod across reschedules. This is essential for databases that need stable network identities for replication.
  • Persistent Volumes: PersistentVolume (PV) represents a piece of storage in the cluster (EBS volume, NFS share, local SSD). PersistentVolumeClaim (PVC) is a request for storage by a Pod. StorageClasses enable dynamic provisioning — when a StatefulSet creates a PVC, Kubernetes automatically provisions the underlying storage. VolumeClaimTemplates in StatefulSets create a unique PVC per Pod replica.
  • Headless Services: StatefulSets use headless Services (clusterIP: None) that create DNS records for each Pod individually (e.g., mysql-0.mysql-svc.default.svc.cluster.local), enabling direct communication for database replication and leader election.
  • Operator Pattern: Kubernetes Operators encode domain-specific knowledge for stateful applications. The MySQL Operator, PostgreSQL Operator, or MongoDB Operator automate tasks like backup, restore, scaling, failover, and version upgrades — managing the full lifecycle of the database through custom controllers.
