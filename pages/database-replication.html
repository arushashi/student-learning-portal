Database Replication

Data redundancy and high availability through database copying

Intermediate ⏱ 30 min

Definition

What is Database Replication?

Database replication is the process of copying and maintaining database objects in multiple database systems to provide data redundancy, high availability, and load distribution.

Key Insight: Replication ensures data availability and improves system reliability through multiple copies.

Simple Explanation

Understanding the Concept

Imagine you have an important notebook with all your notes. To protect against losing it, you make copies and keep them in different locations. Database replication works the same way — it keeps identical copies of your database across multiple servers so that if one fails, the others can take over seamlessly.

How Database Replication Works

1. Replication Topologies

  • Master-Slave (Primary-Replica): One primary node handles all writes; replicas receive copies for read operations
  • Multi-Master: Multiple nodes accept writes, with changes propagated to all other nodes
  • Peer-to-Peer: All nodes are equal, each can read and write independently
  • Circular Replication: Each node replicates to the next in a ring topology

2. Replication Methods

  • Synchronous Replication: Waits for all replicas to confirm the write before committing — ensures strong consistency but adds latency
  • Asynchronous Replication: Commits on the primary immediately, replicas catch up later — faster but risks data loss
  • Semi-Synchronous: Waits for at least one replica to acknowledge before committing — balances consistency and performance
  • Logical Replication: Replicates SQL statements or row-level changes rather than raw data blocks

Replication Challenges

Replication Lag: Delay between primary writes and replica updates causing stale reads

Conflict Resolution: Handling simultaneous writes to different masters in multi-master setups

Split Brain: Network partitions causing nodes to operate independently with divergent data

Real-World Example

MySQL Replication at Facebook

Facebook uses MySQL replication extensively to handle billions of queries per day. Their setup involves primary databases that handle writes with numerous read replicas distributed across data centers globally, ensuring low-latency reads for users worldwide.

Key Points & Summary

Sync vs Async

Synchronous replication guarantees consistency but adds latency; asynchronous offers speed but risks temporary inconsistency.

Master-Slave

The most common topology with one write node and multiple read replicas for scalable read performance.

Replication Lag

The delay between a write on the primary and its appearance on replicas can cause stale data reads.

Conflict Resolution

Multi-master setups need strategies like last-write-wins, merge, or custom conflict handlers.

Quick Summary

  • Replication creates and maintains copies of data across multiple servers
  • Master-slave is the most widely used replication topology
  • Synchronous replication ensures consistency; asynchronous improves performance
  • Replication lag can cause stale reads and must be managed
  • Multi-master replication requires conflict resolution strategies

Interview Questions & Answers

What are the different types of database replication?

Types include: Master-slave (one primary, multiple secondaries), Master-master (multiple primaries), Multi-source (multiple masters to one slave), and Circular replication. Each type has different use cases and trade-offs.

What is the difference between synchronous and asynchronous replication?

Synchronous replication ensures data consistency by waiting for confirmation from replicas before committing, while asynchronous replication provides better performance by allowing immediate commits with eventual consistency.

How do you handle replication lag in distributed systems?

Strategies for handling replication lag include:

  • Read-after-write consistency: Route reads to the primary for recently written data
  • Monotonic reads: Ensure a user always reads from the same replica
  • Lag monitoring: Track replication delay and alert when it exceeds thresholds
  • Semi-synchronous replication: Ensure at least one replica is up-to-date before confirming writes

What is split-brain in database replication and how do you prevent it?

Split-brain occurs when network partitions cause multiple nodes to believe they are the primary:

  • Quorum-based decisions: Require majority agreement before promoting a new primary
  • Fencing (STONITH): Shut down the old primary to prevent dual-write scenarios
  • Leader election: Use consensus algorithms like Raft or Paxos for reliable leader selection
  • Heartbeat monitoring: Detect failures quickly and trigger coordinated failover
