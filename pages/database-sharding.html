Database Sharding

Horizontal partitioning of data across multiple database instances for massive scalability and performance

Definition

What is Database Sharding?

Database sharding is a horizontal partitioning strategy that distributes data across multiple independent database instances, called shards. Each shard holds a subset of the total data and operates as a fully functional database. Unlike vertical partitioning (which splits columns), sharding splits rows — for example, users with IDs 1–1,000,000 go to Shard A, while users 1,000,001–2,000,000 go to Shard B. This allows a system to scale beyond the limits of a single database server by spreading reads, writes, and storage across multiple machines.

Key Insight: Sharding is a last-resort scaling strategy. It introduces significant complexity in application logic, cross-shard queries, and data rebalancing. Always exhaust vertical scaling, read replicas, caching, and query optimization before sharding.

Simple Explanation

Understanding the Concept

Imagine a massive library with 10 million books. A single building can't hold them all efficiently — finding any book takes too long because the catalog is enormous. So you split the library into 10 branches: Branch A holds books by authors A–C, Branch B holds D–F, and so on. Each branch has its own catalog, shelves, and librarians. If someone wants a book by "Garcia," they go straight to Branch C. This is sharding — splitting a large dataset into smaller, manageable pieces distributed across separate instances.

In database systems, sharding becomes necessary when a single database server cannot handle the data volume or request rate. The process involves:

  • Choosing a shard key: The column or attribute used to determine which shard holds each row (e.g., user_id, region, tenant_id)
  • Applying a sharding strategy: Range-based, hash-based, directory-based, or geographic sharding to distribute data evenly
  • Routing queries: A routing layer or application logic directs each query to the correct shard based on the shard key
  • Handling cross-shard operations: Joins, aggregations, and transactions that span multiple shards require special coordination

Real-World Example

Twitter

Twitter shards its tweet storage by user ID. When you post a tweet, it is stored on the shard determined by your user ID using a hash function. This means all tweets from the same user live on the same shard, making user timeline queries fast (single-shard reads). Twitter processes over 500 million tweets per day across thousands of shards. Their shard routing layer uses consistent hashing so that adding new shards only requires migrating a fraction of the data rather than reshuffling everything.

Key Points & Summary

Shard Key Selection

The shard key is the most critical decision in sharding. A good shard key has high cardinality (many unique values), even distribution (no hotspots), and aligns with query patterns (most queries include the shard key). Poor shard key choices lead to unbalanced shards, hotspots, and excessive cross-shard queries. Common shard keys include user_id, tenant_id, and geographic region.

Sharding Strategies

Range-based: Split by value ranges (e.g., dates, IDs) — simple but prone to hotspots. Hash-based: Hash the shard key for even distribution — great balance but loses range query efficiency. Directory-based: A lookup table maps each key to its shard — flexible but the directory is a single point of failure. Geographic: Shard by region for data locality and compliance.

Cross-Shard Challenges

Queries spanning multiple shards are expensive. Cross-shard joins require scatter-gather patterns. Distributed transactions need two-phase commit (2PC) or saga patterns, which are slow and complex. Aggregations (COUNT, SUM) must query all shards and merge results. Unique constraints across shards are nearly impossible to enforce efficiently. Design your schema to minimize cross-shard operations.

Rebalancing & Resharding

As data grows unevenly, shards become imbalanced (some shards store far more data than others). Rebalancing involves migrating data between shards while the system is live — an extremely complex operation. Consistent hashing minimizes data movement when adding or removing shards. Tools like Vitess (for MySQL) and native sharding in MongoDB automate much of this complexity.

Quick Summary

  • Sharding splits data horizontally across multiple database instances (shards)
  • The shard key determines data distribution — choose one with high cardinality and even distribution
  • Common strategies: range-based, hash-based, directory-based, and geographic sharding
  • Cross-shard queries, joins, and transactions are expensive and should be minimized
  • Rebalancing is complex — consistent hashing helps reduce data movement
  • Exhaust simpler scaling strategies (replicas, caching, indexing) before sharding

Sharding Strategies Comparison

Tradeoffs between different sharding approaches

Range-Based: Simple to implement, supports range queries, but prone to hotspots if data is skewed
Hash-Based: Even data distribution, no hotspots, but range queries require querying all shards
Directory-Based: Maximum flexibility, custom routing rules, but lookup table is a bottleneck and SPOF
Geographic: Data locality for compliance and low latency, but uneven distribution across regions

Interview Questions & Answers

When should you shard a database, and what alternatives should you try first?

Sharding should be considered only after simpler scaling strategies have been exhausted:

  • Try first — Vertical scaling: Upgrade to a more powerful server (more RAM, faster SSDs, more CPU cores). This is the simplest approach and works until you hit hardware limits.
  • Try first — Read replicas: For read-heavy workloads, add read replicas to distribute SELECT queries. This can handle 10x read traffic without touching the write path.
  • Try first — Caching: Add Redis or Memcached to cache frequently accessed data. This can reduce database load by 80-90%.
  • Try first — Query optimization: Add proper indexes, fix N+1 queries, denormalize hot tables, and use connection pooling.
  • Shard when: A single database server cannot handle the write volume, the dataset exceeds single-server storage, or you need geographic data partitioning for compliance (e.g., GDPR requires EU user data in EU servers).

How do you choose a good shard key?

Choosing a shard key is the most important and irreversible decision in sharding design:

  • High cardinality: The key should have many distinct values. A boolean field (true/false) would only create 2 shards. user_id or order_id with millions of values enables fine-grained distribution.
  • Even distribution: The key should distribute data uniformly. Sharding by country would put 70% of data on the "US" shard for a US-heavy service. Hashing the key can help with even distribution.
  • Query alignment: Most queries should include the shard key so they can be routed to a single shard. If 90% of queries filter by user_id, then user_id is a natural shard key.
  • Avoid monotonically increasing keys: Using an auto-increment ID with range sharding sends all new writes to the last shard (hotspot). Use hash-based sharding or compound keys instead.
  • Consider compound shard keys: Combining tenant_id + timestamp can provide both data isolation and temporal distribution.

How do you handle cross-shard queries and transactions?

Cross-shard operations are the biggest challenge of sharding and should be minimized by design:

  • Scatter-gather pattern: For queries that must touch multiple shards, send the query to all relevant shards in parallel, then merge results at the application layer. This works for aggregations but is expensive.
  • Denormalization: Duplicate data across shards to avoid cross-shard joins. For example, store the user's name alongside each order so you don't need to join the users shard with the orders shard.
  • Saga pattern: Instead of distributed transactions (2PC), use a saga — a sequence of local transactions on each shard with compensating actions if any step fails. This sacrifices strict consistency for availability.
  • Global tables: Small, rarely-changing reference data (countries, currencies, categories) can be replicated to all shards to avoid cross-shard lookups.
  • Application-level joins: Fetch data from multiple shards independently and join them in the application code. Less efficient but more flexible than database-level joins.

How does consistent hashing relate to database sharding?

Consistent hashing solves the rebalancing problem that arises when shards are added or removed:

  • The problem with naive hashing: If you use shard = hash(key) % N, then changing N (adding a shard) causes nearly all keys to map to different shards, requiring massive data migration.
  • How consistent hashing works: Both shard identifiers and data keys are mapped onto a circular hash ring. Each key is assigned to the next shard clockwise on the ring. When a shard is added, only keys between the new shard and its predecessor need to move — roughly 1/N of the data.
  • Virtual nodes: Each physical shard is represented by multiple virtual nodes on the ring to ensure even distribution. Without virtual nodes, shards can end up with vastly unequal portions of the ring.
  • Real-world usage: Amazon DynamoDB, Apache Cassandra, and Discord all use consistent hashing for shard assignment. Vitess (YouTube's MySQL sharding layer) uses a similar concept with key ranges.
  • Tradeoff: Consistent hashing minimizes data movement but adds complexity to the routing layer and makes range queries impossible (since the hash scrambles key ordering).
