Database Indexing

Database indexing strategies for query optimization and efficient data retrieval

Intermediate ⏱ 30 min

Definition

What is Database Indexing?

A database index is a data structure that improves the speed of data retrieval operations on a database table at the cost of additional storage space and slower writes. Indexes work like a book's table of contents — instead of scanning every page (full table scan), you look up the topic in the index to find the exact page number. Indexes are built on one or more columns of a table using structures like B-trees, hash tables, or inverted indexes, enabling the database engine to locate rows without examining every record in the table.

Key Insight: Indexing is the single most impactful optimization for database query performance. A well-designed index can reduce query time from minutes to milliseconds, but over-indexing can degrade write performance and waste storage. The art lies in choosing the right indexes for your query patterns.

Simple Explanation

Understanding the Concept

Imagine a library with millions of books but no catalog system. To find a specific book, you'd have to walk through every aisle and check every shelf — that's a full table scan. Now imagine the library has a card catalog organized alphabetically by title, author, and subject. You look up the card, find the exact shelf location, and go straight to the book. That card catalog is an index.

Database indexes work the same way. They maintain a sorted reference structure that points directly to the data rows matching your query criteria. Different index types serve different query patterns, and choosing the right one is crucial for performance.

  • B-Tree Indexes: The default index type in most databases (PostgreSQL, MySQL). A self-balancing tree structure that maintains sorted data and supports range queries, equality lookups, and prefix searches efficiently with O(log n) complexity
  • Hash Indexes: Use a hash function to map keys to locations. Excellent for exact equality lookups with O(1) average time, but cannot support range queries or sorting. Ideal for columns used exclusively in WHERE equality conditions
  • Composite Indexes: Indexes on multiple columns that follow the leftmost prefix rule. An index on (country, city, zip) supports queries filtering by country alone, country+city, or all three — but not city alone or zip alone
  • Covering Indexes: Indexes that include all columns needed by a query, allowing the database to satisfy the query entirely from the index without accessing the table data (index-only scan), dramatically improving performance

Real-World Example

Google Search — Inverted Indexes

Google indexes billions of web pages using inverted indexes, which map every word to the list of documents containing that word. When you search for "database indexing," Google doesn't scan every page on the internet. Instead, it looks up "database" and "indexing" in its inverted index, finds the intersection of documents containing both terms, ranks them using PageRank and other signals, and returns results in milliseconds. This same inverted index approach powers Elasticsearch, Apache Solr, and full-text search in PostgreSQL.

Key Points & Summary

B-Tree vs Hash Indexes

B-tree indexes store data in a sorted, balanced tree with O(log n) lookups. They support equality, range queries, ORDER BY, and prefix matching. Hash indexes offer O(1) equality lookups but cannot handle ranges or sorting. Most databases default to B-tree because of its versatility across diverse query patterns.

Composite Indexes & Column Order

The order of columns in a composite index matters enormously. Place the most selective (highest cardinality) columns first and follow the leftmost prefix rule. An index on (A, B, C) supports queries on A, (A, B), and (A, B, C) but not B alone or C alone. Reordering columns based on query patterns can eliminate the need for multiple separate indexes.

Covering Indexes

A covering index includes all columns a query needs in the index itself. The database can return results directly from the index without touching the table data (an "index-only scan"). This eliminates random I/O to the heap, which can improve query performance by 10-100x for frequently executed queries.

Index Trade-offs

Every index speeds up reads but slows down writes because each INSERT, UPDATE, or DELETE must also update the index. Indexes consume additional disk space (often 10-30% of table size). Over-indexing a write-heavy table can severely degrade performance. Use EXPLAIN/ANALYZE to verify that indexes are actually being used by the query planner.

Quick Summary

  • Indexes are data structures that speed up data retrieval at the cost of storage and write performance
  • B-tree indexes are the default and support equality, range, and sort queries with O(log n) time
  • Hash indexes excel at equality lookups with O(1) time but cannot handle range queries
  • Composite index column order follows the leftmost prefix rule and should match query patterns
  • Covering indexes eliminate table lookups by including all needed columns in the index
  • Always use EXPLAIN ANALYZE to verify index usage and avoid over-indexing write-heavy tables

Interview Questions & Answers

When should you use a B-tree index versus a hash index?

The choice depends on your query patterns:

  • B-Tree Index: Use when you need range queries (WHERE age > 25), ORDER BY, LIKE prefix matching (LIKE 'John%'), or a mix of equality and range conditions. B-trees are the default and most versatile index type.
  • Hash Index: Use when you only need exact equality lookups (WHERE email = 'user@example.com'). Hash indexes are faster for equality (O(1) vs O(log n)) but cannot support ranges, sorting, or partial matches.
  • Practical Note: In PostgreSQL, hash indexes became crash-safe only in version 10. In MySQL InnoDB, B-tree is the only option. Most production systems use B-tree indexes almost exclusively due to their flexibility.

How do you decide the column order in a composite index?

Column order in a composite index should follow these guidelines:

  • Equality First: Place columns used in equality conditions (=) before those used in range conditions (>, <, BETWEEN). The index can only use a range condition on the last column it traverses.
  • Selectivity: Among equality columns, consider placing higher-cardinality columns first for better filtering, though this is less critical than the equality-before-range rule.
  • Query Coverage: Design composite indexes to support your most frequent queries. An index on (status, created_at) serves both WHERE status = 'active' and WHERE status = 'active' AND created_at > '2024-01-01'.
  • Avoid Redundancy: An index on (A, B) makes a separate index on (A) alone unnecessary, since the composite index covers single-column lookups on A.

What is a covering index and when would you use one?

A covering index includes all columns that a query needs, allowing the database to satisfy the query entirely from the index without accessing the actual table rows:

  • Example: For query SELECT name, email FROM users WHERE status = 'active', an index on (status, name, email) is a covering index.
  • Benefit: Eliminates random I/O to the heap/clustered index. The query reads only the compact index pages, which are often cached in memory.
  • Performance: Can improve query speed by 10-100x for large tables, especially on spinning disks where random I/O is expensive.
  • Trade-off: Wider indexes consume more space and slow down writes more. Use covering indexes for your highest-frequency, most performance-critical queries.

What are the downsides of having too many indexes?

Over-indexing causes several problems:

  • Write Penalty: Every INSERT must update all indexes. Every UPDATE to an indexed column must remove the old entry and add a new one. On a table with 10 indexes, a single row insert triggers 10 index updates.
  • Storage Overhead: Each index can consume 10-30% of the table size. Unused indexes waste valuable disk space and memory in the buffer pool.
  • Query Planner Confusion: Too many indexes can cause the optimizer to choose suboptimal execution plans, especially with stale statistics.
  • Maintenance Overhead: Indexes need periodic maintenance (REINDEX, VACUUM in PostgreSQL). More indexes mean longer maintenance windows and more lock contention.
  • Solution: Regularly audit indexes using pg_stat_user_indexes (PostgreSQL) or sys.dm_db_index_usage_stats (SQL Server) and drop unused indexes.
