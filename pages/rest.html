REST & RESTfulness

API design principles and architectural style for web services

Intermediate ⏱ 25 min
1 Definition
2 Explanation
3 Key Points

5 Quiz

Definition

What is REST?

REST (Representational State Transfer) is an architectural style for designing networked applications, using HTTP methods to operate on resources.

Key Insight: REST uses stateless communication and standard HTTP methods for scalable API design.

Simple Explanation

Understanding the Concept

Think of a library. Each book has a unique call number (URL), and there are standard actions you can perform: borrow a book (GET), donate a new book (POST), replace a damaged copy (PUT), or remove a book (DELETE). REST works the same way — every resource has a unique address and a fixed set of operations.

Core REST Principles

1. Client-Server Separation

  • The client (frontend) and server (backend) are independent
  • They communicate only through a well-defined API
  • Either side can evolve independently without breaking the other

2. Statelessness

  • Each request from client to server must contain all information needed
  • The server does not store client session state between requests
  • This makes the system easier to scale horizontally

3. Uniform Interface

  • Resources are identified by URIs (e.g., /api/users/42)
  • Standard HTTP methods: GET, POST, PUT, PATCH, DELETE
  • Representations (JSON, XML) transfer resource state
  • HATEOAS provides navigation through hypermedia links

HTTP Methods in REST

GET — Retrieve a resource (idempotent, safe)

POST — Create a new resource

PUT — Replace an entire resource (idempotent)

PATCH — Partially update a resource

DELETE — Remove a resource (idempotent)

HTTP Status Codes

  • 200 OK: Successful request
  • 201 Created: Resource successfully created
  • 204 No Content: Successful with no response body
  • 400 Bad Request: Invalid client request
  • 401 Unauthorized: Authentication required
  • 404 Not Found: Resource does not exist
  • 500 Internal Server Error: Server-side failure

Real-World Example

Twitter's REST API

Twitter's API follows REST conventions: GET /2/tweets/:id retrieves a tweet, POST /2/tweets creates one, and DELETE /2/tweets/:id removes it. Each endpoint operates on a clearly identified resource using standard HTTP methods, making the API predictable and easy to integrate with.

Key Points & Summary

Statelessness

Every request carries all context needed for processing, making servers easy to scale and replace without session affinity.

Resource-Based URLs

Resources are identified by clean, hierarchical URIs such as /users/42/orders, making APIs intuitive and self-documenting.

Idempotency

GET, PUT, and DELETE are idempotent — calling them multiple times produces the same result, enabling safe retries.

Cacheability

REST responses can be marked cacheable or non-cacheable, reducing server load and improving client performance.

Quick Summary

  • REST stands for Representational State Transfer
  • Uses standard HTTP methods to operate on resources
  • Stateless communication enables horizontal scaling
  • HATEOAS enables discoverability through hypermedia links
  • Proper status codes communicate request outcomes clearly

Interview Questions & Answers

What are the principles of REST?

REST principles include: Client-server architecture, Statelessness, Cacheability, Uniform interface, Layered system, and Code on demand (optional). These principles ensure scalability and simplicity.

What is the difference between REST and RESTful?

REST is the architectural style, while RESTful describes an API that follows REST principles. A RESTful API properly implements HTTP methods, status codes, and resource-based URLs.

What is idempotency and why does it matter in REST?

Idempotency means making the same request multiple times produces the same result. It matters because:

  • Safe retries: Network failures can cause duplicate requests; idempotent methods handle this gracefully
  • GET, PUT, DELETE: These are idempotent by design in REST
  • POST: Not idempotent — each call may create a new resource
  • Idempotency keys: Used with POST to make non-idempotent operations safe

How do you version a REST API?

Common API versioning strategies include:

  • URI versioning: /api/v1/users — simplest and most visible
  • Query parameter: /api/users?version=1
  • Header versioning: Custom header like X-API-Version: 1
  • Content negotiation: Accept: application/vnd.myapi.v1+json

URI versioning is the most common approach because it's simple and easy to discover.
