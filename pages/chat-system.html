Chat System Design

Designing a real-time chat system like WhatsApp or Slack with message delivery guarantees

Definition

What is a Chat System?

A real-time chat system is a platform that enables instant messaging between users through persistent, bidirectional communication channels. It supports one-on-one conversations, group chats, media sharing, read receipts, and online presence indicators. At scale, services like WhatsApp handle over 100 billion messages per day across 2 billion users, requiring sophisticated architectures for real-time message delivery, reliable offline message storage, end-to-end encryption, and global distribution. Designing a chat system is one of the most complex system design challenges because it combines real-time communication, eventual consistency, and massive scale.

Key Insight: The fundamental challenge of a chat system is maintaining the illusion of instant, reliable communication over an inherently unreliable network. Every message must be delivered exactly once, in the correct order, even when users go offline, switch networks, or are on opposite sides of the globe.

Simple Explanation

Understanding the Concept

Think of a chat system like a postal service with superpowers. In the traditional postal system, you write a letter, drop it at the post office, and the carrier delivers it to the recipient's mailbox. If the recipient isn't home, the letter waits in the mailbox. A chat system works similarly but at lightning speed: your message goes to a central server, which instantly pushes it to the recipient if they're online, or stores it for later delivery if they're offline.

The "superpowers" come from WebSockets — a technology that keeps a persistent, two-way connection between each user and the server. Unlike traditional HTTP where the client must constantly ask "any new messages?", WebSockets allow the server to instantly push messages to clients the moment they arrive, enabling true real-time communication.

  • WebSocket Connections: Each online user maintains a persistent WebSocket connection to a chat server. Messages are pushed instantly through this connection, achieving sub-100ms delivery latency. The server tracks which users are connected to which server instances for routing
  • Message Ordering: Messages within a conversation must be displayed in the correct order. This is achieved using server-assigned timestamps or logical clocks (Lamport timestamps), with sequence numbers per conversation to handle clock skew between servers
  • Offline Message Delivery: When a recipient is offline, messages are stored in a persistent queue. Upon reconnection, the client requests all messages with sequence numbers greater than its last received message, ensuring no messages are lost
  • Presence System: Tracking which users are online, offline, or idle requires a heartbeat mechanism. Clients send periodic pings; if no heartbeat is received within a timeout (e.g., 30 seconds), the user is marked offline and their contacts are notified

Real-World Example

WhatsApp — 100B+ Messages Per Day

WhatsApp handles over 100 billion messages daily with a remarkably lean engineering team. Their architecture uses Erlang/OTP on FreeBSD for the messaging servers, chosen for its exceptional concurrency model where each user connection is a lightweight Erlang process. Messages are stored temporarily on the server and deleted once delivered (store-and-forward model). WhatsApp uses the Signal Protocol for end-to-end encryption, meaning even WhatsApp servers cannot read message content. Group messages use a fan-out-on-write approach where the sender's server distributes copies to each group member's message queue.

Key Points & Summary

Real-Time Delivery

WebSockets provide full-duplex, persistent connections for instant message delivery with sub-100ms latency. Long polling serves as a fallback for environments where WebSockets are blocked. Server-Sent Events (SSE) work for one-way notifications but lack bidirectional support needed for chat. Each chat server maintains a connection registry mapping user IDs to active WebSocket connections.

Message Ordering & Delivery

Messages use server-assigned monotonically increasing sequence numbers per conversation to guarantee ordering. Delivery follows a three-state model: sent (server received), delivered (recipient's device received), and read (recipient opened the chat). Each state change triggers an acknowledgment back through the WebSocket, powering the familiar single-check, double-check, blue-check indicators.

Presence System

Online presence is tracked via heartbeats sent every 5-10 seconds over the WebSocket connection. When a heartbeat is missed beyond a configurable timeout (typically 30 seconds), the user is marked offline. Presence updates are propagated to relevant contacts using a pub/sub system (Redis Pub/Sub), but updates are throttled and batched to avoid overwhelming the system with status changes for users with thousands of contacts.

Group Chat Fan-Out

When a user sends a group message, the server must deliver it to all group members. Small groups (under 500 members) use fan-out-on-write: the message is immediately copied to each member's inbox queue. Large groups or channels use fan-out-on-read: the message is stored once, and members fetch it when they open the group. This hybrid approach balances write amplification against read latency.

Quick Summary

  • WebSockets enable bidirectional, real-time message delivery with persistent connections
  • Message ordering uses server-assigned sequence numbers per conversation
  • Three delivery states: sent, delivered, read — each confirmed via acknowledgments
  • Offline messages are queued and synced on reconnection using sequence-based catch-up
  • Presence tracking uses heartbeats with pub/sub propagation to online contacts
  • Group chats use fan-out-on-write for small groups and fan-out-on-read for large channels

Interview Questions & Answers

Why use WebSockets instead of HTTP polling for a chat application?

WebSockets are superior to HTTP polling for chat for several reasons:

  • Latency: WebSockets deliver messages instantly as they arrive. HTTP polling introduces delay equal to the polling interval (even 1-second polling adds average 500ms latency).
  • Efficiency: A WebSocket connection has minimal overhead after the initial handshake. HTTP polling sends full HTTP headers with every request, wasting bandwidth — at scale this means billions of unnecessary requests per day.
  • Bidirectional: WebSockets allow the server to push data to the client without the client asking. HTTP is strictly request-response, requiring the client to constantly poll.
  • Scalability: A single server can maintain millions of idle WebSocket connections with minimal CPU usage, whereas the same number of polling clients would overwhelm the server with request processing.
  • Fallback: Long polling (HTTP request that blocks until data is available) serves as a reliable fallback when WebSockets are blocked by firewalls or proxies.

How do you ensure messages are delivered in the correct order?

Message ordering in a distributed chat system is achieved through:

  • Per-Conversation Sequence Numbers: Each conversation maintains a monotonically increasing sequence number. When a message is stored, it's assigned the next sequence number atomically. Clients display messages sorted by sequence number.
  • Server-Side Timestamps: Use the server's timestamp rather than the client's to avoid clock skew issues. A centralized or synchronized time source (NTP, Google TrueTime) provides consistent ordering.
  • Handling Concurrent Messages: When two users send messages simultaneously, the server serializes them using a per-conversation lock or optimistic concurrency control, ensuring a deterministic order.
  • Client-Side Reordering: Clients may receive messages out of order due to network delays. The client buffers incoming messages and reorders them by sequence number before displaying, requesting missing sequence numbers if gaps are detected.

How would you design the presence (online/offline) system?

A presence system tracks and broadcasts user online status:

  • Heartbeat Mechanism: Each connected client sends a heartbeat ping every 5-10 seconds. The server stores the last heartbeat timestamp in Redis with a TTL. If the TTL expires (no heartbeat received), the user is considered offline.
  • Status Propagation: When a user's status changes, publish the event to a Redis Pub/Sub channel. Only servers holding WebSocket connections to the user's contacts subscribe to these channels, avoiding broadcasting to the entire cluster.
  • Optimization for Large Friend Lists: For users with thousands of contacts, batch and throttle presence updates. A user who rapidly toggles between online and offline (flapping) should have their status updates debounced to prevent notification storms.
  • Lazy Propagation: Instead of pushing status to all contacts proactively, fetch presence status on-demand when a user opens a chat window. This dramatically reduces the number of presence updates for users with large contact lists.

How do you handle message delivery when the recipient is offline?

Offline message delivery requires a store-and-forward architecture:

  • Message Queue per User: Each user has a persistent inbox queue (stored in Cassandra or a similar database). When the recipient is offline, the message is written to their queue and a push notification is sent via APNs/FCM.
  • Sync on Reconnection: When a user comes online, the client sends its last received sequence number. The server returns all messages with sequence numbers greater than the client's last known position, ensuring no messages are missed.
  • Message Expiry: WhatsApp's store-and-forward model keeps messages on the server only until delivered, then deletes them. Other systems retain messages permanently for cross-device sync (like Telegram/Slack).
  • Push Notifications: For high-priority messages, send push notifications through platform-specific channels (Apple Push Notification Service, Firebase Cloud Messaging) to wake the app and prompt the user to reconnect.
