System Reliability

High availability and fault tolerance principles

Definition

What is System Reliability?

System reliability is the probability that a system will perform its intended function without failure over a specified period under stated conditions.

Key Insight: Reliability is achieved through redundancy, fault tolerance, and proactive monitoring.

Simple Explanation

Understanding the Concept

Think of a hospital's power supply. It has the main grid, a backup generator, and battery-powered UPS units. If one fails, the next takes over instantly because lives depend on it. System reliability works the same way — layers of safeguards ensure services keep running even when individual components fail.

Key Reliability Concepts

1. Reliability Metrics

  • MTBF (Mean Time Between Failures): Average time a system operates before failing
  • MTTR (Mean Time To Recovery): Average time to restore service after a failure
  • Availability: Percentage of uptime (e.g., 99.99% = 52.6 min downtime/year)
  • Error rate: Percentage of requests that result in errors

2. Fault Tolerance Patterns

  • Circuit Breaker: Stops cascading failures by halting requests to failing services
  • Retry with Backoff: Automatically retries failed operations with increasing delays
  • Bulkhead: Isolates components so one failure doesn't sink the whole system
  • Failover: Automatically switches to a standby system on failure

3. Redundancy Strategies

  • Active-Active: Multiple instances handle traffic simultaneously
  • Active-Passive: Standby instance takes over when the primary fails
  • Data replication: Copies of data across multiple nodes or regions
  • Geographic redundancy: Deploying across multiple data centers

4. Graceful Degradation

  • Core features continue to work even when non-critical components fail
  • Feature flags disable problematic functionality dynamically
  • Fallback responses serve cached or simplified data
  • Load shedding drops low-priority requests under extreme load

Chaos Engineering

Chaos engineering proactively tests system resilience by intentionally injecting failures — killing servers, introducing network latency, or corrupting data — in controlled experiments. This practice, pioneered by Netflix's Chaos Monkey, reveals weaknesses before they cause real outages.

SLAs, SLOs, and SLIs

  • SLA (Service Level Agreement): A formal contract defining service level commitments and penalties
  • SLO (Service Level Objective): Internal targets for reliability metrics (e.g., 99.95% uptime)
  • SLI (Service Level Indicator): The actual measured metrics (e.g., request latency, error rate)

Real-World Example

Netflix's Resilience Architecture

Netflix pioneered chaos engineering with Chaos Monkey, which randomly terminates production instances to ensure services can withstand failures. Their architecture uses circuit breakers (Hystrix), bulkheads, and multi-region failover so that even if an entire AWS region goes down, viewers can continue streaming without interruption.

Key Points & Summary

Health Checks

Continuous health monitoring detects failures early, enabling automatic recovery and traffic rerouting before users are impacted.

Circuit Breakers

Circuit breakers prevent cascading failures by quickly failing requests to unhealthy services instead of waiting for timeouts.

Redundancy

Running multiple instances across regions ensures no single point of failure can bring the entire system down.

Chaos Engineering

Proactively injecting failures in controlled experiments reveals weaknesses and builds confidence in system resilience.

Quick Summary

  • MTBF and MTTR are key metrics for measuring reliability
  • Circuit breakers, retries, and bulkheads prevent cascading failures
  • Graceful degradation maintains core features during partial outages
  • Chaos engineering proactively tests system resilience
  • SLAs define formal service level commitments with customers

Interview Questions & Answers

How do you measure system reliability?

Reliability is measured using metrics like Mean Time Between Failures (MTBF), Mean Time To Recovery (MTTR), availability percentage (99.9%, 99.99%), and error rates. These metrics help quantify system dependability.

What are common reliability patterns?

Common patterns include circuit breakers, retry mechanisms, bulkheads, failover systems, redundancy, health checks, and graceful degradation. These patterns help maintain service availability during failures.

What is the difference between availability and reliability?

Availability and reliability are related but distinct concepts:

  • Availability: The proportion of time a system is operational and accessible (uptime vs total time)
  • Reliability: The probability of a system functioning correctly without failure over a period
  • Key distinction: A system can be highly available (quick recovery) but unreliable (frequent failures), or highly reliable (rare failures) but with planned downtime
  • Both matter: Aim for high reliability to minimize failures and high availability to minimize impact when they occur

How does a circuit breaker work in distributed systems?

A circuit breaker monitors calls to a downstream service and operates in three states:

  • Closed (normal): Requests pass through; failures are counted
  • Open (tripped): When failures exceed a threshold, all requests fail immediately without calling the downstream service
  • Half-Open (testing): After a timeout, a limited number of requests are allowed through to test if the service has recovered
  • Benefits: Prevents cascading failures, reduces load on failing services, and provides fast failure feedback to callers
