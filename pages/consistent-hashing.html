Consistent Hashing

A distributed hashing technique that minimizes key remapping when nodes are added or removed from the system

Advanced ⌚ 25 min 📅 April 2, 2024

Definition

What is Consistent Hashing?

Consistent hashing is a distributed hashing scheme that maps both data keys and server nodes onto a virtual ring (hash space from 0 to 2^32 - 1). Each key is assigned to the nearest server node found by walking clockwise around the ring. When a node is added or removed, only approximately 1/N of the keys need to be remapped (where N is the number of nodes), unlike traditional modular hashing where nearly all keys must be redistributed.

Key Insight: Consistent hashing ensures that when a server is added or removed, only a minimal fraction of keys (~1/N) need to move, making it ideal for distributed caches, databases, and CDNs where rehashing the entire dataset would be catastrophically expensive.

Simple Explanation

Understanding the Concept

Imagine a circular clock face where both your data items and your servers are placed at various positions around the clock. Each data item travels clockwise from its position until it finds the nearest server — that server is responsible for storing that data. If a server is removed, its data simply moves to the next server clockwise. If a server is added, it only takes responsibility for a small arc of data from the next server clockwise.

With traditional modular hashing (key % N), adding or removing a single server changes where almost every key maps. Consistent hashing solves this with a hash ring approach:

  • Hash Ring: Both keys and nodes are hashed onto a circular space (0 to 2^32 - 1). Each key is assigned to the first node encountered clockwise.
  • Minimal Disruption: Adding a node only affects keys between the new node and its predecessor; removing a node only moves keys to the successor node.
  • Virtual Nodes: Each physical node is mapped to multiple positions on the ring (e.g., 150-200 virtual nodes per physical server) to ensure even distribution of keys.
  • Scalability: Systems can grow or shrink incrementally without massive data migration, enabling elastic scaling under varying loads.

Real-World Example

Amazon DynamoDB

Amazon DynamoDB uses consistent hashing to partition data across its storage nodes. When DynamoDB scales out by adding new nodes, only a fraction of the key space is migrated. Each physical node is represented by multiple virtual nodes on the hash ring, ensuring balanced data distribution across the cluster even as nodes join and leave. This design, originally described in the Amazon Dynamo paper (2007), allows DynamoDB to handle millions of requests per second while dynamically adjusting capacity.

Key Points & Summary

The Hash Ring

Both keys and server nodes are hashed onto a circular address space (0 to 2^32 - 1). Keys are assigned to the first server found clockwise from their hash position. This circular mapping eliminates the modular dependency on total node count.

Virtual Nodes (Vnodes)

Each physical server is mapped to many positions on the ring. This solves the problem of uneven key distribution that arises when few physical nodes create large gaps. More powerful servers can have more virtual nodes to receive a proportionally larger share of the data.

Minimal Redistribution

When a node joins or leaves, only approximately K/N keys need to be remapped (K = total keys, N = total nodes). This is dramatically better than modular hashing, where nearly all keys must move. Data migration becomes proportional rather than catastrophic.

Load Balancing

With a sufficient number of virtual nodes (typically 100-200 per server), consistent hashing achieves near-uniform distribution. The standard deviation of load per node decreases as virtual node count increases, providing predictable and balanced storage utilization.

Quick Summary

  • Consistent hashing maps keys and servers onto a virtual ring for stable assignment
  • Only ~1/N keys need remapping when a node is added or removed
  • Virtual nodes ensure balanced distribution across heterogeneous servers
  • Used in DynamoDB, Cassandra, Memcached, and CDN routing
  • Eliminates the complete rehashing problem of traditional key % N approaches
  • Supports heterogeneous clusters by assigning more virtual nodes to powerful servers

Interview Questions & Answers

Why is consistent hashing preferred over simple modular hashing in distributed systems?

With simple modular hashing (key % N), adding or removing a server changes N, causing almost every key to map to a different server. This means massive data migration. Consistent hashing solves this because:

  • Only ~K/N keys (where K is total keys) need to be remapped when a node changes
  • The hash ring structure decouples key assignment from total node count
  • Supports smooth elastic scaling without downtime or heavy data migration
  • Avoids cache stampede or hot-spot issues that arise from full redistribution
  • Provides deterministic key assignment without centralized coordination

What are virtual nodes and why are they necessary?

Virtual nodes (vnodes) are multiple hash positions assigned to a single physical server on the consistent hash ring. They solve critical problems:

  • Uneven Distribution: With few physical nodes, the hash space is split unevenly. Virtual nodes create many more points on the ring, smoothing out distribution.
  • Heterogeneous Hardware: Powerful servers can be assigned more virtual nodes, receiving a proportionally larger data share.
  • Better Recovery: When a node fails, its load is spread across many other nodes (each virtual node's data goes to a different successor), avoiding overloading a single server.
  • Typical Count: Systems like DynamoDB use 100-200 virtual nodes per physical server to achieve near-uniform distribution.

How does consistent hashing handle node failures and additions?

Node Failure:

  • When a node fails, all keys assigned to it are automatically re-routed to the next node clockwise on the ring
  • With virtual nodes, the failed node's keys are distributed across multiple successors rather than one
  • Combined with replication (keys stored on N successive nodes), data loss is prevented

Node Addition:

  • A new node takes over a portion of keys from its successor on the ring
  • Only the keys that now fall in the new node's range need to be transferred
  • The process is incremental and does not disrupt the rest of the cluster

Name real-world systems that use consistent hashing and explain how.

Several major distributed systems rely on consistent hashing:

  • Amazon DynamoDB: Uses consistent hashing with virtual nodes to partition data across storage nodes for scalable key-value storage
  • Apache Cassandra: Assigns token ranges on a consistent hash ring; each node owns a portion of the ring and replicates data to subsequent nodes
  • Memcached (libketama): Client libraries use consistent hashing to distribute cache keys across memcached servers, minimizing cache misses during scaling
  • Akamai CDN: Uses consistent hashing to route content requests to the nearest/most appropriate edge server with minimal reshuffling
  • Discord: Uses consistent hashing to distribute user sessions and messages across their infrastructure
