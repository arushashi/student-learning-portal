Definition

What is System Performance?

System performance refers to the ability of a software system to process requests efficiently, minimize response times, and maximize throughput under varying loads. It encompasses a broad set of optimization techniques including latency reduction, resource utilization, caching strategies, connection pooling, query optimization, and profiling. Performance engineering ensures that applications meet their Service Level Objectives (SLOs) by measuring key metrics like P50, P95, and P99 response times, requests per second (RPS), and error rates.

Key Insight: Performance is not just about speed — it is about delivering a consistently fast experience at scale. A system with low average latency but terrible P99 latency can still frustrate users, because 1 in 100 requests will be painfully slow.

Simple Explanation

Understanding the Concept

Think of system performance like a highway. Latency is how long it takes a single car to travel from point A to point B. Throughput is how many cars can pass through the highway per hour. A four-lane highway has higher throughput than a two-lane road, but each car's travel time (latency) might remain the same. Performance optimization means widening the highway, removing bottlenecks, and ensuring smooth traffic flow.

In software systems, every user request travels through multiple layers — DNS resolution, network hops, load balancers, application servers, databases, and back. Performance optimization targets each of these layers to reduce end-to-end response time:

  • Latency reduction: Minimize the time each request spends waiting — through caching, connection reuse, and efficient algorithms
  • Throughput maximization: Increase the number of requests the system can handle per second via concurrency, batching, and horizontal scaling
  • Profiling & benchmarking: Use tools like flame graphs, APM agents, and load testing to identify bottlenecks before they hit production
  • Resource optimization: Tune CPU, memory, disk I/O, and network utilization to avoid waste and prevent saturation

Real-World Example

Google Search

Google returns search results in under 200 milliseconds, querying billions of web pages across thousands of servers worldwide. They achieve this through aggressive caching of popular queries, pre-computed indexes, connection pooling to backend services, geographic distribution of data centers, and constant profiling to shave milliseconds off every layer of the stack. Google's engineers famously found that adding just 500ms of latency to search results reduced traffic by 20%.

Key Points & Summary

Latency vs Throughput

Latency measures the time to complete a single request (e.g., 50ms API response). Throughput measures total work done per unit time (e.g., 10,000 RPS). Optimizing one doesn't always improve the other — batching can increase throughput at the cost of individual request latency. Understanding this tradeoff is critical for designing SLOs.

Profiling & Benchmarking

Never optimize without measuring first. Use Application Performance Monitoring (APM) tools like Datadog, New Relic, or Jaeger for distributed tracing. Flame graphs reveal CPU hotspots. Load testing with tools like k6, Locust, or JMeter simulates production traffic to find breaking points before users do.

Caching as #1 Optimization

Caching is the single most effective performance optimization. Use in-memory caches like Redis or Memcached for frequently accessed data. Apply caching at every layer: browser cache, CDN cache, application cache, and database query cache. A well-placed cache can reduce database load by 90% and cut response times from 200ms to under 5ms.

Connection Pooling & Batching

Creating new database or HTTP connections is expensive (TCP handshake, TLS negotiation). Connection pools maintain a set of reusable connections, eliminating setup overhead. Batching groups multiple small operations into a single round-trip — for example, fetching 100 user records in one query instead of 100 individual queries (the N+1 problem).

Quick Summary

  • Latency is per-request time; throughput is total requests per second — optimize both
  • Always measure before optimizing: use profiling, tracing, and load testing
  • Caching is the most impactful optimization — apply it at every layer
  • Connection pooling eliminates expensive connection setup overhead
  • Percentile metrics (P95, P99) reveal tail latency that averages hide
  • Query optimization (indexes, denormalization, avoiding N+1) is essential for database-bound systems

Interview Questions & Answers

How would you diagnose a slow API endpoint in production?

A systematic approach to diagnosing a slow API endpoint involves multiple layers of investigation:

  • Check metrics dashboards: Start with APM tools (Datadog, New Relic) to see latency percentiles (P50, P95, P99), error rates, and throughput trends over time
  • Distributed tracing: Use Jaeger or Zipkin to trace a single slow request through all microservices and identify which service or database call is the bottleneck
  • Database query analysis: Check for slow queries using EXPLAIN plans, look for missing indexes, N+1 query patterns, or full table scans
  • Resource utilization: Examine CPU, memory, disk I/O, and network metrics on the host — a saturated resource will cause queuing delays
  • Connection pool exhaustion: Verify that database and HTTP connection pools are not fully utilized, causing requests to wait for available connections
  • External dependencies: Check if third-party API calls or downstream services are responding slowly, and consider adding circuit breakers and timeouts

When should you scale vertically vs horizontally to improve performance?

The choice between vertical and horizontal scaling depends on the nature of the bottleneck:

  • Vertical scaling (scale up): Add more CPU, RAM, or faster disks to a single machine. Best for: database servers with complex queries, applications not designed for distribution, or when the bottleneck is single-threaded processing. Simpler but has hardware limits and creates a single point of failure.
  • Horizontal scaling (scale out): Add more machines behind a load balancer. Best for: stateless application servers, read-heavy workloads (add read replicas), or when you need fault tolerance. Requires the application to handle distributed state, session management, and data consistency.
  • Hybrid approach: Most production systems use both — scale up database servers to the practical limit, then shard horizontally. Scale out stateless app servers behind load balancers from the start.
  • Cost considerations: Vertical scaling has diminishing returns (2x CPU rarely gives 2x performance). Horizontal scaling is more cost-effective at scale but adds operational complexity.

What are P99 percentiles and why do they matter more than averages?

Percentile metrics provide a much more accurate picture of user experience than averages:

  • P99 definition: The P99 latency is the value below which 99% of all requests complete. If your P99 is 500ms, then 1 in 100 requests takes longer than 500ms.
  • Why averages lie: An average of 50ms could mean most requests take 30ms but a few take 2000ms. The average hides the pain of those slow requests.
  • Impact at scale: If a user's page load makes 50 API calls, there is a 40% chance (1 - 0.99^50) that at least one call hits P99 latency. High-percentile latency directly impacts real user experience.
  • SLO design: Industry best practice is to set SLOs on P95 and P99, not averages. For example: "P99 latency must be below 300ms" is a meaningful SLO.
  • Common causes of tail latency: Garbage collection pauses, cache misses, retries, network timeouts, and resource contention under load.

Explain connection pooling and why it improves performance.

Connection pooling is a technique that maintains a pool of reusable connections to avoid the overhead of creating new ones:

  • The problem: Every new database connection requires a TCP handshake (1 RTT), TLS negotiation (1-2 RTT), and authentication. This can add 5-30ms per connection, which is devastating at high request rates.
  • How pooling works: A pool of pre-established connections is maintained. When the application needs a connection, it borrows one from the pool, uses it, and returns it. No setup overhead per request.
  • Pool sizing: Set the pool size based on expected concurrency. Too small causes request queuing; too large wastes database resources. A common formula: connections = (core_count * 2) + effective_spindle_count.
  • Connection pool tools: HikariCP (Java), pgBouncer (PostgreSQL), ProxySQL (MySQL), and built-in pools in ORMs like SQLAlchemy or Sequelize.
  • Beyond databases: HTTP connection pooling (keep-alive) and gRPC channel reuse apply the same principle to service-to-service communication.
