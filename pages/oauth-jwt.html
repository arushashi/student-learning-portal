OAuth 2.0 & JWT

Learn the OAuth 2.0 authorization framework and JSON Web Token (JWT) based authentication — the industry standards for secure delegated access and stateless token-based authentication in modern web applications and APIs.

Definition

What are OAuth 2.0 & JWT?

OAuth 2.0 is an authorization framework (defined in RFC 6749) that enables third-party applications to obtain limited access to a user's resources on another service without exposing the user's credentials. Instead of sharing passwords, OAuth uses access tokens — short-lived credentials that specify what the third-party app can do (scopes) and for how long. For example, when you click "Sign in with Google" on a third-party website, OAuth 2.0 allows that website to access your Google profile information without ever knowing your Google password.

JSON Web Token (JWT, pronounced "jot", defined in RFC 7519) is a compact, self-contained token format used to securely transmit information between parties as a JSON object. A JWT consists of three Base64URL-encoded parts separated by dots: header.payload.signature. The header specifies the signing algorithm, the payload contains claims (user ID, roles, expiration time), and the signature ensures the token hasn't been tampered with. JWTs are commonly used as the access token format in OAuth 2.0 implementations, and they enable stateless authentication — the server can verify the token's authenticity and extract user information without querying a database or session store.

Key Insight: OAuth 2.0 and JWT solve different but complementary problems. OAuth 2.0 is an authorization framework that defines the flow for obtaining tokens — how a user grants a third-party app permission to access their resources. JWT is a token format that defines how those tokens are structured and verified. You can use OAuth 2.0 without JWT (using opaque tokens that require server-side validation), and you can use JWT without OAuth (for simple API authentication). However, combining them is the most common modern pattern: OAuth 2.0 handles the authorization dance, and JWTs serve as the access tokens, enabling stateless, scalable authentication across microservices.

Detailed Explanation

Understanding the Concept

Think of OAuth 2.0 like a hotel key card system. When you check into a hotel (the authorization server), you prove your identity with your ID (credentials). The hotel gives you a key card (access token) that grants access to specific rooms (resources) for a specific duration (token expiry). You can give a copy of this key card to a friend (third-party app) so they can access the gym (limited scope) without giving them your actual identity documents. The hotel can also revoke the key card at any time. JWT is like the technology inside the key card — it encodes your room number, access level, and check-out date in a tamper-proof format that any door lock (resource server) can read and verify without calling the front desk (authorization server) every time.

  • OAuth 2.0 Authorization Code Flow: The most secure and common OAuth flow for web applications. Step 1: The app redirects the user to the authorization server's login page with a client_id, redirect_uri, scope, and state parameter. Step 2: The user authenticates and grants permission. Step 3: The authorization server redirects back to the app with a short-lived authorization code. Step 4: The app's backend exchanges this code (plus client_secret) for an access token and optionally a refresh token. The authorization code acts as a one-time intermediary — it's exchanged server-to-server, so the access token never passes through the user's browser, preventing interception. PKCE (Proof Key for Code Exchange) extends this flow for public clients like mobile apps that can't securely store a client_secret.
  • JWT Structure — Header, Payload, Signature: A JWT like eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiIxMjM0NTY3ODkwIn0.signature has three parts. The Header contains the token type ("JWT") and signing algorithm (e.g., RS256, HS256). The Payload contains claims — registered claims like sub (subject/user ID), exp (expiration), iat (issued at), iss (issuer); and custom claims like roles, permissions, or tenant_id. The Signature is created by signing the encoded header and payload with a secret key (HMAC) or private key (RSA/ECDSA). Any resource server holding the corresponding secret/public key can verify the token's integrity without contacting the authorization server, enabling stateless validation across distributed microservices.
  • Access Tokens vs Refresh Tokens: Access tokens are short-lived (typically 15-60 minutes) and sent with every API request in the Authorization header (Bearer <token>). Their short lifespan limits the damage if intercepted. Refresh tokens are long-lived (days to months) and stored securely — they're used exclusively to obtain new access tokens when the current one expires, without requiring the user to re-authenticate. The refresh flow: when an API returns 401 (access token expired), the client sends the refresh token to the authorization server's token endpoint and receives a new access token. Refresh tokens should be rotated on each use (the old one is invalidated and a new one is issued) to detect token theft — if a stolen refresh token is used after the legitimate user has already rotated it, the authorization server detects the reuse and revokes all tokens for that session.
  • Security Best Practices: Always use HTTPS for all OAuth flows to prevent token interception. Store access tokens in memory (JavaScript variable) rather than localStorage or cookies vulnerable to XSS. For refresh tokens in browsers, use httpOnly, secure, sameSite cookies. Validate JWT claims on every request: check exp (not expired), iss (trusted issuer), aud (intended audience), and signature. Use asymmetric signing (RS256/ES256) so resource servers only need the public key — they can verify tokens without the ability to create them. Implement token revocation for logout using a short-lived token allowlist or a token deny list cached in Redis for real-time revocation. Never store sensitive data in JWT payloads — they're only Base64-encoded, not encrypted.

Real-World Example

"Sign in with Google" on Third-Party Apps

When you click "Sign in with Google" on a website like Medium, Figma, or Notion, you're experiencing OAuth 2.0 in action. Here's what happens: The third-party app redirects your browser to Google's authorization server (accounts.google.com) with parameters specifying what information it needs (scope: email, profile). You log in to Google (if not already) and see a consent screen: "Medium wants to access your email address and profile picture — Allow?" When you click "Allow," Google redirects you back to Medium with an authorization code. Medium's backend server exchanges this code with Google for an access token (and often an ID token, which is a JWT containing your identity claims via OpenID Connect). Medium uses the access token to call Google's userinfo API to get your name, email, and profile picture, then creates or links a Medium account. At no point did Medium see your Google password. If Medium is compromised, attackers only get access tokens with limited scopes — not your Google credentials. You can revoke Medium's access anytime from your Google account security settings. This pattern is used by millions of applications worldwide, with Google, Facebook, GitHub, Apple, and Microsoft serving as identity providers through OAuth 2.0 and OpenID Connect.

Key Points

OAuth 2.0 Grant Types

OAuth 2.0 defines multiple grant types for different client types. Authorization Code (with PKCE) is the recommended flow for web and mobile apps — the most secure option using a server-side code exchange. Client Credentials is used for machine-to-machine communication where no user is involved (e.g., a cron job calling an API) — the client authenticates with its own credentials and receives a token directly. Implicit grant (now deprecated in OAuth 2.1) returned tokens directly in the redirect URL, exposing them to the browser history and referrer headers. Device Authorization Grant is used for input-limited devices (smart TVs, CLI tools) — the device displays a code that the user enters on a separate device with a browser. Resource Owner Password Credentials (also deprecated) allowed direct username/password exchange, undermining the delegated authorization model.

JWT Structure & Validation

A JWT contains three dot-separated Base64URL-encoded segments: header, payload, and signature. Validation requires checking multiple aspects: (1) Decode the header and verify the signing algorithm matches what you expect (preventing algorithm confusion attacks where an attacker switches from RS256 to HS256 using the public key as the HMAC secret). (2) Verify the signature using the issuer's public key (for RSA/ECDSA) or shared secret (for HMAC). (3) Check the exp claim to ensure the token hasn't expired. (4) Validate the iss (issuer) matches your trusted authorization server. (5) Verify the aud (audience) includes your service to prevent tokens meant for other services from being accepted. JWTs are not encrypted by default — use JWE (JSON Web Encryption) if the payload contains sensitive data.

Access vs Refresh Tokens

Access tokens and refresh tokens serve complementary roles in a secure token lifecycle. Access tokens are sent with every API request, have short lifespans (15-60 minutes), and contain the user's identity and permissions. Their brevity limits the window of exploitation if stolen. Refresh tokens are stored securely and only sent to the authorization server's token endpoint — never to resource servers. They have longer lifespans (days to weeks) and are used solely to obtain fresh access tokens. Refresh token rotation (issuing a new refresh token with each use and invalidating the old one) provides token theft detection: if both the attacker and legitimate user try to use the same refresh token, the server detects the reuse and invalidates the entire token family, forcing re-authentication. This pattern provides the security benefits of short-lived tokens without the UX penalty of frequent re-logins.

Security Best Practices

Securing OAuth and JWT implementations requires defense in depth. Always use HTTPS (TLS 1.2+) for all OAuth endpoints and token transmission. Use the state parameter to prevent CSRF attacks in the authorization code flow. Implement PKCE for all public clients (mobile apps, SPAs) to prevent authorization code interception. Store tokens appropriately: access tokens in memory for SPAs, refresh tokens in httpOnly/secure/sameSite cookies. Use asymmetric signing (RS256 or ES256) so resource servers can verify but not forge tokens. Implement token revocation for logout — maintain a deny list of revoked JWT IDs (jti claim) in Redis with TTL matching the token's remaining lifetime. Set restrictive scopes — request only the minimum permissions needed. Validate the redirect_uri parameter exactly (no wildcard matching) to prevent open redirect attacks.

Quick Summary

  • OAuth 2.0 is an authorization framework for delegated access; JWT is a self-contained token format — they complement each other
  • The Authorization Code flow with PKCE is the recommended grant type for web and mobile applications
  • JWTs contain header (algorithm), payload (claims), and signature (tamper protection) in a base64-encoded dot-separated format
  • Short-lived access tokens limit exposure; refresh tokens provide seamless token renewal without re-authentication
  • Refresh token rotation detects token theft by invalidating the entire token family on reuse detection
  • Always validate JWT issuer, audience, expiration, and signature; use asymmetric signing to separate token creation from verification

Interview Questions

Q1: Explain the OAuth 2.0 Authorization Code flow. Why is it more secure than the Implicit flow?

The Authorization Code flow adds a critical server-side exchange step that keeps tokens out of the browser:

  • Flow Steps: (1) The client redirects the user to the authorization server with client_id, redirect_uri, scope, state, and code_challenge (PKCE). (2) The user authenticates and consents. (3) The authorization server redirects back with a short-lived authorization code (valid for ~60 seconds, single-use). (4) The client's backend server exchanges the code plus client_secret (or code_verifier for PKCE) for an access token and refresh token via a direct server-to-server HTTPS call.
  • Security Advantage: The access token is never exposed to the browser — the authorization code passes through the browser's redirect, but it's useless without the client_secret/code_verifier. Even if the code is intercepted (e.g., in browser history or server logs), the attacker can't exchange it because they don't have the client_secret.
  • Implicit Flow Weaknesses: The Implicit flow returned the access token directly in the URL fragment (#access_token=...), exposing it to browser history, referrer headers, and JavaScript on the page. It provided no refresh tokens, so users had to re-authenticate frequently. It's now deprecated in OAuth 2.1.
  • PKCE Extension: For public clients (mobile apps, SPAs) that can't store a client_secret, PKCE (Proof Key for Code Exchange) adds a dynamically generated code_verifier/code_challenge pair that proves the client exchanging the code is the same one that initiated the flow, preventing authorization code interception attacks.

Q2: How do you handle JWT token revocation since JWTs are stateless?

JWT revocation is challenging because the stateless nature of JWTs means there's no central session store to invalidate. Several strategies address this:

  • Short Expiration + Refresh Tokens: The simplest approach — use very short-lived access tokens (5-15 minutes). Even if a token is compromised, it expires quickly. Combine with refresh tokens that can be revoked server-side. When a user logs out, revoke their refresh token so no new access tokens can be issued. The tradeoff: there's a window (up to the token's remaining lifetime) where a revoked token is still technically valid.
  • Token Deny List: Maintain a deny list (in Redis) of revoked JWT IDs (jti claim). On each request, the resource server checks if the JWT's jti is in the deny list. Set Redis TTL equal to the token's remaining lifetime so entries auto-expire. This adds a fast cache lookup to each request but provides immediate revocation. The deny list stays small since entries expire.
  • Token Versioning: Store a "token version" per user in the database. Include this version in the JWT. When revoking (e.g., password change, security breach), increment the user's token version. Resource servers compare the JWT's version with the stored version — mismatches mean the token is revoked.
  • Hybrid Approach: Use stateless JWT validation for most requests (fast, no network call) but check the deny list for sensitive operations (payment, account changes). This balances performance with security granularity.

Q3: What is the difference between authentication and authorization? How do OAuth 2.0 and OpenID Connect address each?

Authentication and authorization are distinct security concerns that are often confused:

  • Authentication (AuthN): Verifying who you are — proving your identity. "Are you really John Doe?" Methods include passwords, biometrics, multi-factor authentication, and certificates. Authentication answers the question of identity.
  • Authorization (AuthZ): Determining what you're allowed to do — checking permissions after identity is established. "John Doe is authenticated, but can he access the admin panel?" Authorization answers the question of permissions and scopes.
  • OAuth 2.0 is for Authorization: OAuth 2.0 is explicitly an authorization framework. It defines how a client obtains permission to access resources on behalf of a user. The access token represents authorization (what the client can do), not authentication (who the user is). OAuth 2.0 alone doesn't tell you who the user is — it tells you what they've permitted.
  • OpenID Connect (OIDC) adds Authentication: OIDC is a thin identity layer built on top of OAuth 2.0. It adds an ID token (always a JWT) that contains identity claims — who the user is (sub, name, email). When you "Sign in with Google," OIDC provides the user's identity via the ID token, while OAuth 2.0 provides authorization via the access token. OIDC also defines a standard userinfo endpoint and standard scopes (openid, profile, email).

Q4: How would you implement JWT-based authentication across a microservices architecture?

JWT-based authentication in microservices requires careful token issuance, propagation, and validation:

  • Centralized Auth Service: A dedicated authentication/authorization service issues JWTs upon successful login. It signs tokens with a private key (RS256/ES256) and publishes the corresponding public key via a JWKS (JSON Web Key Set) endpoint. All other services can fetch this public key to verify tokens without needing the private key or direct communication with the auth service.
  • API Gateway Validation: The API gateway validates the JWT on every incoming request — checking signature, expiration, issuer, and audience. Invalid tokens are rejected at the edge before reaching internal services. The gateway can extract user claims and forward them as headers to internal services, eliminating the need for each service to re-validate the full JWT.
  • Token Propagation: When Service A calls Service B internally, it forwards the JWT in the Authorization header. Service B can trust the claims without re-authentication. For service-to-service calls not initiated by users (e.g., async jobs, event handlers), use the Client Credentials grant to obtain machine-to-machine tokens with appropriate scopes.
  • Key Rotation: Periodically rotate signing keys using the JWKS endpoint. Include a kid (key ID) claim in the JWT header so resource servers know which key to use for verification. During rotation, both old and new keys are available in the JWKS endpoint, ensuring existing tokens remain valid while new tokens use the new key.
