Idempotency

Operations that produce the same result regardless of how many times they are executed

Intermediate ⏱ 25 min

Definition

What is Idempotency?

Idempotency is a property of certain operations in mathematics and computer science where performing the operation multiple times produces the same result as performing it once. In distributed systems, an idempotent operation guarantees that no matter how many times a request is executed — whether due to retries, network failures, or duplicate submissions — the system state and response remain identical to a single successful execution. This concept is fundamental to building reliable, fault-tolerant APIs and distributed systems.

Key Insight: Idempotency is what allows you to safely press the "Pay Now" button twice without being charged double. It transforms unreliable networks into reliable systems by making retries inherently safe, which is the cornerstone of resilient distributed architectures.

Simple Explanation

Understanding the Concept

Think of an elevator button. No matter how many times you press the "3rd floor" button, the elevator goes to the 3rd floor exactly once. Pressing it ten times doesn't send you to the 30th floor. That's idempotency in everyday life — the result is the same whether you perform the action once or a hundred times.

In distributed systems, network failures are inevitable. A client sends a payment request, but the connection drops before it receives a confirmation. Did the payment go through? The client doesn't know, so it retries. Without idempotency, this retry could charge the customer twice. With idempotency, the server recognizes the duplicate request and returns the original result.

  • Idempotency Keys: Unique tokens attached to requests that allow servers to detect and deduplicate repeated operations, commonly implemented as UUIDs sent in request headers
  • At-Least-Once Delivery: Message systems guarantee delivery but may send duplicates; idempotent consumers handle this safely by deduplicating on the receiver side
  • Exactly-Once Semantics: The holy grail of messaging achieved by combining at-least-once delivery with idempotent processing, ensuring each message is effectively processed once
  • Retry Safety: Idempotent operations can be retried freely without side effects, enabling automatic retry policies with exponential backoff for resilient distributed communication

Real-World Example

Stripe Payment API

Stripe uses idempotency keys to prevent double charges. When a client sends a payment request, it includes an Idempotency-Key header with a unique UUID. If the network fails and the client retries with the same key, Stripe recognizes the duplicate and returns the original charge result instead of creating a new one. Stripe stores these keys for 24 hours, ensuring that any retries within that window are safely deduplicated. This mechanism processes billions of dollars in transactions without accidental double charges.

Key Points & Summary

Definition & Importance

Idempotency ensures that repeating an operation produces the same outcome as a single execution. In distributed systems where network partitions, timeouts, and retries are inevitable, idempotency is critical for maintaining data consistency and preventing unintended side effects like duplicate payments or orders.

Idempotency Keys

Idempotency keys are unique client-generated tokens (typically UUIDs) sent with each request. The server stores the key alongside the response. On subsequent requests with the same key, the server returns the cached response instead of re-executing the operation. Keys typically expire after 24-48 hours.

HTTP Method Idempotency

GET, PUT, and DELETE are inherently idempotent by the HTTP specification. GET retrieves data without modification. PUT replaces the entire resource with the same result each time. DELETE removes a resource, and deleting it again has no additional effect. POST is not idempotent as it creates new resources with each call.

Database Operations

Database idempotency is achieved through upserts (INSERT ON CONFLICT UPDATE), conditional updates (UPDATE WHERE version = X), and unique constraints. Using optimistic concurrency control with version numbers ensures that replayed writes don't corrupt data or create duplicates.

Quick Summary

  • Idempotent operations yield identical results regardless of execution count
  • GET, PUT, and DELETE are idempotent; POST is not by default
  • Idempotency keys allow servers to deduplicate retried requests
  • Exactly-once semantics combine at-least-once delivery with idempotent processing
  • Database upserts, version checks, and unique constraints enforce idempotency at the storage layer
  • Idempotency is essential for payment systems, order processing, and any operation with financial impact

Interview Questions & Answers

What is idempotency and why is it important in distributed systems?

Idempotency means that performing an operation multiple times has the same effect as performing it once. It is critical in distributed systems because:

  • Network Unreliability: Messages can be lost, duplicated, or delayed. Clients often need to retry requests without knowing if the original succeeded.
  • Retry Safety: Idempotent operations can be safely retried with exponential backoff, enabling robust error recovery patterns.
  • Consistency: Without idempotency, duplicate requests could create duplicate records, charge customers twice, or corrupt system state.
  • Exactly-Once Semantics: Combining at-least-once delivery with idempotent consumers achieves the practical equivalent of exactly-once processing.

How would you implement idempotency in a payment processing system?

A robust payment idempotency implementation involves:

  • Idempotency Key: Require clients to send a unique key (UUID) with each payment request in a header like Idempotency-Key.
  • Key Storage: Store the key, request hash, and response in a fast store like Redis with a TTL of 24-48 hours.
  • Atomic Check-and-Execute: Use a distributed lock or database transaction to atomically check for key existence and execute the payment if new.
  • Request Fingerprint: Hash the request body and compare it on retry to detect misuse where different payloads share the same idempotency key.
  • In-Flight Handling: If a request with the same key is still being processed, return a 409 Conflict or block until the original completes.

Which HTTP methods are idempotent and which are not? Explain why.

Per the HTTP specification (RFC 7231):

  • GET (Idempotent): Only reads data. Multiple identical GET requests return the same response without changing server state.
  • PUT (Idempotent): Replaces the entire resource. Sending the same PUT twice still results in the same resource state.
  • DELETE (Idempotent): Removes a resource. Deleting an already-deleted resource returns 404 but doesn't alter state further.
  • HEAD (Idempotent): Same as GET but returns only headers. Safe and idempotent.
  • POST (Not Idempotent): Creates a new resource each time. Two identical POST requests may create two separate resources.
  • PATCH (Depends): Setting a field to a value is idempotent; incrementing a counter is not. It depends on the operation semantics.

What is the difference between at-least-once, at-most-once, and exactly-once delivery?

These are message delivery guarantees in distributed systems:

  • At-Most-Once: Messages are delivered zero or one time. No retries on failure. Simple but may lose messages. Used when occasional loss is acceptable (e.g., metrics, logs).
  • At-Least-Once: Messages are delivered one or more times. Retries ensure delivery but may produce duplicates. Requires idempotent consumers to handle safely.
  • Exactly-Once: Messages are delivered and processed exactly one time. Theoretically impossible in truly distributed systems (per the Two Generals Problem), but achievable in practice by combining at-least-once delivery with idempotent processing and transactional outbox patterns.

Most production systems use at-least-once delivery with idempotent consumers, as it provides the best balance of reliability and simplicity.
