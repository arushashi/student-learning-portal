Event Sourcing

Store application state as an immutable sequence of events, enabling complete audit trails, temporal queries, and state reconstruction

Definition

What is Event Sourcing?

Event Sourcing is an architectural pattern where the state of an application is determined by a sequence of immutable events rather than by storing just the current state. Instead of updating a record in place (as in traditional CRUD), every state change is captured as an event object and appended to an append-only event store. The current state of any entity can be reconstructed by replaying all its events from the beginning. Events are facts that happened in the past and are never modified or deleted, providing a complete and immutable history of everything that has occurred in the system.

Key Insight: Think of your bank account statement. Instead of just showing your current balance ($1,250), the bank keeps a ledger of every transaction: deposited $2,000, withdrew $500, deposited $100, paid bill $350. The current balance is derived by replaying all transactions. If you dispute a charge, the bank can trace exactly what happened and when. That ledger is event sourcing.

Simple Explanation

Understanding the Concept

In traditional systems, when you update a customer's address, you overwrite the old address with the new one. The old address is lost forever. With Event Sourcing, you store an "AddressChanged" event containing both the old and new address, along with a timestamp and who made the change. The current address is determined by replaying all events, and the full history of address changes is always available.

Core concepts of Event Sourcing:

  • Event Store: An append-only log where all events are persisted. Events are immutable once stored. The event store is the single source of truth. Technologies like EventStoreDB, Apache Kafka, or even a relational database table can serve as an event store.
  • Events as Facts: Each event represents something that happened in the past: "OrderPlaced," "PaymentProcessed," "ItemShipped." Events are named in past tense because they describe facts that have already occurred. They contain all the data needed to describe the change.
  • State Reconstruction: The current state of an aggregate (entity) is rebuilt by loading all its events from the event store and applying them in sequence. This process is called "rehydration." For example, an Order's state is rebuilt by replaying OrderCreated, ItemAdded, ItemRemoved, and OrderConfirmed events.
  • Projections: Since querying an event stream directly is impractical for most use cases, projections (also called read models or materialized views) are built by processing events. A projection subscribes to events and maintains a denormalized, query-optimized view of the data. Multiple projections can be built from the same event stream.
  • Snapshots: For aggregates with many events, replaying all events can be slow. Snapshots periodically capture the current state so that reconstruction only needs to replay events after the last snapshot, improving performance.

Key benefits of Event Sourcing:

  • Complete Audit Trail: Every change is recorded with who, what, when, and why, providing full traceability
  • Temporal Queries: You can reconstruct the state of any entity at any point in time by replaying events up to that moment
  • Debugging: Replay events to reproduce bugs and understand exactly how the system reached a particular state
  • New Projections: Add new read models by replaying the complete event history, extracting insights you didn't anticipate when the system was designed

Real-World Example

Banking and Financial Systems

Banks are the quintessential example of event sourcing. A bank account's balance is never stored directly; instead, the bank records every transaction as an event: deposits, withdrawals, transfers, interest payments, and fees. The current balance is the result of replaying all these events. This approach is legally required for financial auditing, enables dispute resolution by examining the exact sequence of events, supports regulatory reporting by projecting transaction data into compliance views, and allows temporal queries like "What was the account balance at close of business on March 15th?" The entire financial industry operates on event sourcing principles through double-entry bookkeeping, which has been used for over 500 years.

Key Points & Summary

Event Store vs State Store

A traditional state store overwrites data with each update, losing history. An event store appends immutable events, preserving the complete history. The event store is the source of truth; current state is derived by replaying events. Use technologies like EventStoreDB, Kafka, or DynamoDB Streams.

Projections

Projections are read-optimized views built by processing events from the event store. They can be rebuilt from scratch at any time by replaying all events. Multiple projections can serve different query needs from the same event stream, each denormalized for its specific use case.

Replay Capability

The ability to replay events enables powerful capabilities: rebuild read models, fix bugs by replaying corrected logic, migrate to new schemas, create new projections retroactively, and perform temporal debugging to understand exact system behavior at any past point.

Audit Trail

Event sourcing provides a built-in, tamper-evident audit trail. Every change is recorded with full context: what changed, when, who triggered it, and why. This is invaluable for compliance (GDPR, SOX, HIPAA), debugging, analytics, and dispute resolution.

Quick Summary

  • Event Sourcing stores state as an append-only sequence of immutable events
  • Current state is derived by replaying all events for an entity (aggregate)
  • Projections build read-optimized views from the event stream for efficient queries
  • Provides complete audit trail, temporal queries, and replay/debugging capabilities
  • Pairs naturally with CQRS for separate read and write models

Interview Questions & Answers

What is Event Sourcing and how does it differ from traditional state-based persistence?

Event Sourcing stores every state change as an immutable event in an append-only log, while traditional persistence stores only the current state by overwriting previous values. Key differences:

  • History: Event Sourcing preserves the complete history of all changes; traditional persistence loses history with each update
  • Storage: Event Sourcing uses an append-only event store; traditional uses mutable tables with UPDATE and DELETE operations
  • Current State: In Event Sourcing, current state is derived by replaying events; in traditional persistence, it's read directly from the database
  • Audit: Event Sourcing has a built-in audit trail; traditional systems need separate audit tables or change data capture
  • Temporal Queries: Event Sourcing naturally supports "what was the state at time X" queries; traditional systems cannot answer this without separate versioning

What are projections in Event Sourcing and why are they needed?

Projections (also called read models or materialized views) are query-optimized data structures built by processing events from the event store. They are needed because:

  • Query Efficiency: Querying an event stream directly is slow and impractical for most use cases. A projection pre-computes and denormalizes data for fast retrieval.
  • Multiple Views: Different consumers need different views of the same data. One projection might power a dashboard, another a search index, and a third an analytics report, all from the same events.
  • Rebuild Capability: If a projection's schema changes or a bug is found, you can delete the projection and rebuild it from scratch by replaying all events through the corrected projection logic.
  • Technology Flexibility: Each projection can use the best storage technology for its use case: SQL for relational queries, Elasticsearch for search, Redis for caching, etc.

Projections are typically updated asynchronously by subscribing to the event stream, making them eventually consistent with the event store.

How do you handle the growing size of event streams? What are snapshots?

As event streams grow, replaying all events to rebuild state becomes slow. Strategies to manage this include:

  • Snapshots: Periodically save the current aggregate state as a snapshot. When loading an aggregate, start from the latest snapshot and replay only events that occurred after it. For example, snapshot every 100 events.
  • Event Stream Partitioning: Partition events by aggregate ID so each stream remains manageable. An order with 50 events loads quickly even if the system has millions of events total.
  • Archiving: Move old, rarely accessed events to cold storage while keeping recent events in hot storage for fast access.
  • Projection Optimization: Since most reads hit projections rather than the event store, optimize projection storage and indexing for your query patterns.

In practice, most aggregates have relatively short event streams (tens to hundreds of events), so performance is rarely an issue without snapshots.

What are the challenges and trade-offs of Event Sourcing?

Event Sourcing introduces several challenges that teams must address:

  • Event Schema Evolution: Events are immutable, but your domain model evolves. You need versioning strategies: upcasting (transforming old event formats to new), weak schema (tolerating unknown fields), or event migration.
  • Eventual Consistency: Projections are updated asynchronously, so reads may lag behind writes. Design your UI and business logic to handle this staleness gracefully.
  • Complexity: The mental model is different from CRUD. Developers must think in terms of events and state derivation rather than direct state manipulation.
  • GDPR and Data Deletion: Events are immutable, but regulations like GDPR require the ability to delete personal data. Solutions include crypto-shredding (encrypting personal data with per-user keys and deleting the key) or event tombstones.
  • Testing: Testing requires constructing event histories, verifying projections, and testing event replay scenarios, which requires different testing approaches.
