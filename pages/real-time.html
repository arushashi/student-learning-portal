Real-Time Communication

WebSockets & WebRTC for instant data exchange

Definition

What is Real-Time Communication?

Real-time communication enables instant data exchange between clients and servers using technologies like WebSockets and WebRTC for live interactions.

Key Insight: Real-time communication eliminates polling delays and provides instant bidirectional messaging.

Simple Explanation

Understanding the Concept

Imagine you're at a live sports event. You see everything happen the instant it occurs — no delay, no waiting. Real-time communication in software works the same way: data flows immediately between parties without the need to constantly ask "is there anything new?"

How Real-Time Communication Works

1. WebSockets

  • An HTTP handshake upgrades the connection to a persistent TCP socket
  • Both client and server can send messages at any time (full-duplex)
  • Low overhead per message compared to repeated HTTP requests
  • Ideal for chat apps, live dashboards, and collaborative editing

2. Server-Sent Events (SSE)

  • One-directional channel from server to client over HTTP
  • Automatic reconnection built into the browser API
  • Great for news feeds, stock tickers, and notification streams
  • Simpler than WebSockets when only server push is needed

3. Long Polling

  • Client sends a request; server holds it open until data is available
  • Once a response is sent, the client immediately opens a new request
  • Works everywhere HTTP works — no special protocol needed
  • Higher latency and overhead than WebSockets

4. WebRTC

  • Peer-to-peer communication directly between browsers
  • Supports audio, video, and arbitrary data channels
  • Uses STUN/TURN servers for NAT traversal
  • Powers video conferencing, screen sharing, and P2P file transfer

Pub/Sub Pattern in Real-Time Systems

The publish/subscribe pattern decouples message producers from consumers. Publishers emit events to topics, and subscribers receive only the events they're interested in. This pattern is fundamental to scalable real-time architectures like chat rooms, live notifications, and event-driven microservices.

Real-World Example

Slack's Messaging Infrastructure

Slack uses WebSockets to deliver messages instantly across millions of channels. When you type a message in a channel, it travels through Slack's gateway servers over a persistent WebSocket connection and is pushed to every connected client in that channel within milliseconds — no page refresh or polling required.

Key Points & Summary

Full-Duplex

WebSockets enable simultaneous two-way communication, letting both client and server send data independently at any time.

Low Latency

Persistent connections eliminate the overhead of establishing new connections for each message, achieving sub-millisecond delivery.

Scalability Patterns

Pub/sub, message brokers, and fan-out strategies help real-time systems scale to millions of concurrent connections.

Peer-to-Peer

WebRTC allows direct browser-to-browser communication for audio, video, and data without routing through a central server.

Quick Summary

  • WebSockets provide persistent, full-duplex communication channels
  • SSE offers simple one-directional server-to-client streaming
  • Long polling simulates real-time over standard HTTP
  • WebRTC enables peer-to-peer media and data exchange
  • Pub/sub patterns decouple producers and consumers for scalability

Interview Questions & Answers

What's the difference between WebSockets and HTTP?

WebSockets provide persistent, bidirectional communication while HTTP is request-response. WebSockets maintain open connections for real-time messaging, reducing overhead compared to HTTP polling.

What is WebRTC used for?

WebRTC enables peer-to-peer real-time communication for video/audio calls, file sharing, and data transfer directly between browsers without server mediation.

How does long polling differ from WebSockets?

Long polling works by holding an HTTP request open until the server has data to send, then the client immediately re-requests. Key differences include:

  • Connection overhead: Long polling creates a new HTTP request each cycle; WebSockets maintain one persistent connection
  • Latency: WebSockets deliver messages faster since the connection is always open
  • Direction: Long polling is essentially server-to-client; WebSockets are full-duplex
  • Compatibility: Long polling works with any HTTP infrastructure; WebSockets require proxy/load-balancer support

How would you scale a WebSocket-based chat system?

Scaling WebSocket systems involves several strategies:

  • Horizontal scaling: Run multiple WebSocket server instances behind a load balancer with sticky sessions
  • Message broker: Use Redis Pub/Sub or Kafka to fan out messages across server instances
  • Connection management: Implement heartbeats and graceful reconnection to handle dropped connections
  • Sharding: Partition users across server groups by chat room or user ID
