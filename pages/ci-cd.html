CI/CD Pipelines

Master Continuous Integration and Continuous Deployment — the automated software delivery practices that enable teams to build, test, and release software changes rapidly, reliably, and repeatedly using build pipelines, automated testing, and deployment strategies.

Definition

What is CI/CD?

CI/CD stands for Continuous Integration and Continuous Delivery/Deployment — a set of software engineering practices and automated pipelines that enable teams to deliver code changes more frequently, reliably, and with confidence. Continuous Integration (CI) is the practice of automatically building and testing code every time a developer merges changes into a shared repository, catching integration bugs early before they compound. Continuous Delivery (CD) extends CI by automatically preparing every successful build for release to production through a series of staging environments, requiring only a manual approval to deploy. Continuous Deployment goes one step further — every change that passes all automated tests is automatically deployed to production without human intervention.

A CI/CD pipeline is the automated workflow that implements these practices: code commit triggers a build, which triggers unit tests, integration tests, security scans, artifact creation, deployment to staging, acceptance tests, and finally deployment to production. Modern CI/CD tools include Jenkins, GitHub Actions, GitLab CI/CD, CircleCI, and AWS CodePipeline. The pipeline ensures that every change goes through the same rigorous, repeatable process, eliminating manual deployment steps that are error-prone and slow.

Key Insight: The distinction between Continuous Delivery and Continuous Deployment is often confused in interviews. Continuous Delivery means every commit is automatically built, tested, and prepared for production release — but deployment requires a human click to approve. Continuous Deployment removes that manual gate entirely — every commit that passes all automated checks is deployed to production automatically. Amazon practices true Continuous Deployment, averaging one production deployment every 11.7 seconds across all their services. The key enabler is not faster pipelines but comprehensive automated testing and deployment strategies (blue-green, canary) that make deploying as safe as not deploying.

Detailed Explanation

Understanding the Concept

Think of CI/CD like a car manufacturing assembly line. In the early days of auto manufacturing, cars were built one at a time by skilled craftsmen — slow, inconsistent, and error-prone. Henry Ford's assembly line revolutionized this: each car moved through standardized stations (chassis, engine, body, paint, quality check) in a predictable, repeatable flow. If a defect was found at the quality check station, the line stopped and the issue was fixed immediately rather than discovering it after 100 more cars were built with the same defect. CI/CD applies the same philosophy to software: every code change moves through an automated assembly line (build → test → stage → deploy), defects are caught immediately at the relevant station, and the entire process is standardized and repeatable regardless of which developer made the change.

  • CI: Build and Test Automation: Continuous Integration begins when a developer pushes code to a shared repository (Git). The CI server (Jenkins, GitHub Actions) detects the change, checks out the code, compiles it (if needed), and runs a comprehensive test suite: unit tests verify individual functions and methods, integration tests verify interactions between components, and static analysis tools (SonarQube, ESLint) check code quality and security vulnerabilities. If any step fails, the developer is immediately notified and the merge is blocked. This "fail fast" approach catches bugs within minutes of introduction, when the context is fresh and the fix is cheap — rather than discovering integration problems days or weeks later.
  • Pipeline Stages — From Commit to Production: A well-designed pipeline typically has these stages: (1) Source — triggered by git push or pull request; (2) Build — compile code, resolve dependencies, create artifacts; (3) Unit Test — run fast, isolated tests (< 5 minutes); (4) Integration Test — test component interactions, database queries, API contracts; (5) Security Scan — SAST (static application security testing), dependency vulnerability scanning, container image scanning; (6) Artifact Publish — push Docker images to registry, store build artifacts; (7) Deploy to Staging — deploy to a production-like environment; (8) Acceptance Test — end-to-end tests, smoke tests, performance tests; (9) Production Deployment — deploy using a safe strategy (blue-green, canary, rolling).
  • Deployment Strategies: How you deploy to production is as important as what you deploy. Blue-Green deployment maintains two identical production environments — "blue" (current) and "green" (new). Traffic is switched from blue to green atomically via DNS or load balancer; if issues arise, you switch back instantly. Canary deployment routes a small percentage of traffic (e.g., 5%) to the new version while the majority stays on the old version. If metrics look good, traffic is gradually shifted. Rolling deployment updates instances incrementally — Kubernetes' default strategy replaces Pods one by one. Feature flags allow deploying code changes without activating them for users, decoupling deployment from release.
  • Rollback Mechanisms: No deployment strategy is complete without a rollback plan. Automated rollback triggers when health checks fail — if error rates spike above a threshold or latency exceeds SLO targets within minutes of deployment, the system automatically reverts to the previous version. Blue-green deployments enable instant rollback by switching the load balancer back. Container-based rollbacks are fast because the previous Docker image is still available in the registry. Database rollbacks are harder — schema migrations should be designed as backwards-compatible (expand-contract pattern) so that the old application version can still work with the new schema, enabling safe rollback without data loss.

Real-World Example

Amazon: Deploying Every 11.7 Seconds

Amazon is the gold standard for Continuous Deployment at scale. Across their thousands of microservices, Amazon averages one production deployment every 11.7 seconds — that's over 7,000 deployments per hour. This is made possible by their fully automated deployment pipeline called Apollo. When an engineer commits code, it automatically progresses through build, unit tests, integration tests, and a series of increasingly realistic test environments called "pipelines." Each service team owns their pipeline configuration, but the tooling is standardized across the organization. Amazon uses a combination of canary deployments and automatic rollback: new versions are first deployed to a single availability zone, and automated systems monitor error rates, latency, and business metrics. If any alarm fires within the bake time (typically 30-60 minutes), the deployment is automatically rolled back without human intervention. This aggressive deployment pace actually makes their systems more reliable — changes are small (often just a few lines of code), easy to understand, easy to debug, and easy to revert. The alternative of deploying large batches of changes infrequently creates "big bang" releases that are risky, hard to debug, and terrifying to rollback.

Key Points

CI vs CD vs CD

Continuous Integration (CI) focuses on automatically building and testing every code change when merged to the main branch, ensuring that integration bugs are caught immediately. Continuous Delivery extends CI by automatically promoting passing builds through staging environments and preparing them for production release — but a human makes the final deployment decision. Continuous Deployment goes further by eliminating the manual gate: every commit that passes all automated quality gates is deployed to production automatically. The maturity progression is: CI (automated build & test) → Continuous Delivery (automated release preparation) → Continuous Deployment (fully automated production deployment). Most organizations start with CI and gradually mature toward Continuous Deployment as their test coverage, monitoring, and rollback capabilities improve.

Pipeline Stages

A CI/CD pipeline is composed of sequential and parallel stages that transform source code into a production deployment. Key stages include: Source (git trigger), Build (compile, dependency resolution), Test (unit, integration, E2E), Security Scan (SAST, DAST, dependency audit), Artifact (Docker image build, push to registry), Deploy to Staging (production-like environment), Acceptance (smoke tests, performance tests, manual QA), and Production Deploy. Stages should be ordered from fastest/cheapest to slowest/most expensive — run unit tests before integration tests, run integration tests before E2E tests. Parallel execution (running tests across multiple agents) reduces total pipeline time. A good CI pipeline completes in under 10 minutes to maintain developer productivity.

Deployment Strategies

Blue-Green maintains two identical environments and switches traffic atomically — zero downtime and instant rollback, but requires double the infrastructure during deployment. Canary releases gradually shift traffic (1% → 5% → 25% → 100%) while monitoring error rates and latency — limits blast radius but is slower to complete. Rolling updates replace instances incrementally (Kubernetes default) — resource-efficient but multiple versions coexist during the update window. A/B testing routes specific user segments to different versions for business metric comparison. Feature flags decouple deployment from release, allowing code to be deployed but not activated until toggled on. The choice depends on risk tolerance, infrastructure cost, and recovery time requirements.

Rollback Mechanisms

Effective rollback is the safety net that makes frequent deployments sustainable. Automated rollback triggers include error rate spikes, latency SLO violations, failed health checks, and anomalous business metrics (sudden drop in conversions or sign-ups). Blue-green rollback is instant — switch the load balancer back to the previous environment. Container-based rollback re-deploys the previous Docker image tag. Database rollbacks require forward-compatible schema migrations: use the expand-contract pattern where you add new columns/tables first, migrate data, update the application, then remove old columns — so the old application version always works with the current schema. Feature flag rollback is the fastest — disable the flag without any deployment. Runbook automation (PagerDuty, Opsgenie) can trigger rollbacks automatically based on alerting rules.

Quick Summary

  • CI automatically builds and tests every code change; CD automates the path to production (Delivery = manual gate, Deployment = fully automated)
  • Pipeline stages progress from fast/cheap checks (lint, unit tests) to slow/expensive ones (E2E tests, security scans, staging deployment)
  • Blue-green deployment provides instant rollback; canary deployment limits blast radius by gradually shifting traffic
  • Automated rollback based on SLO violations is essential for safe Continuous Deployment at scale
  • Feature flags decouple deployment from release, enabling dark launches and gradual feature rollouts
  • Database schema changes should be backwards-compatible to support safe rollback without data loss

Interview Questions

Q1: What is the difference between Continuous Delivery and Continuous Deployment?

While often used interchangeably, these are distinct practices with different levels of automation:

  • Continuous Delivery: Every code change that passes all automated tests is automatically built, tested, and prepared for production release — but the actual deployment to production requires manual approval (a human clicks "deploy"). The software is always in a deployable state, and the team can release at any time with confidence, but they choose when to release based on business timing, regulatory requirements, or coordination needs.
  • Continuous Deployment: Every code change that passes all automated quality gates is automatically deployed to production without any manual intervention. There is no human in the loop between a developer's commit and production. This requires extremely mature testing, monitoring, and automated rollback capabilities to be safe.
  • Trust Gradient: The progression from CI → Continuous Delivery → Continuous Deployment represents increasing trust in automation. Organizations move along this spectrum as they improve test coverage, monitoring, deployment strategies, and incident response automation.
  • In Practice: Most organizations practice Continuous Delivery for user-facing services (with manual approval for production) and Continuous Deployment for internal services, configuration changes, and non-critical systems where the risk tolerance is higher.

Q2: Compare blue-green, canary, and rolling deployment strategies. When would you use each?

Each strategy offers different trade-offs between safety, speed, cost, and complexity:

  • Blue-Green: Maintain two identical production environments. Deploy the new version to the inactive environment, run smoke tests, then switch all traffic via load balancer or DNS. Pros: instant rollback (switch back), zero downtime, simple to understand. Cons: requires double infrastructure cost during deployment, database migrations must be handled carefully. Best for: mission-critical systems where instant rollback is essential and infrastructure cost is acceptable.
  • Canary: Deploy the new version alongside the old one and gradually shift traffic (1% → 5% → 25% → 50% → 100%) while monitoring metrics. If any issues are detected, roll back the canary. Pros: limits blast radius, real production testing, data-driven rollout decisions. Cons: complex routing configuration, multiple versions coexist, slower to complete. Best for: large-scale systems where you need to validate with real traffic before full rollout, especially with diverse user populations.
  • Rolling: Replace instances one at a time (or in small batches). Kubernetes does this by default with maxSurge and maxUnavailable parameters. Pros: resource-efficient (no double infrastructure), built into Kubernetes. Cons: multiple versions coexist during rollout, rollback requires a new rolling update, slower for large deployments. Best for: standard microservice deployments where brief version coexistence is acceptable.
  • Recommendation: Use canary for high-traffic, user-facing services. Use blue-green for databases and stateful services. Use rolling for internal services and batch processing. Combine with feature flags for maximum control.

Q3: How do you handle database schema changes in a CI/CD pipeline?

Database migrations are the hardest part of CI/CD because unlike application deployments, schema changes can't be simply rolled back:

  • Expand-Contract Pattern: Phase 1 (Expand) — add the new column/table alongside the old one. Phase 2 (Migrate) — update the application to write to both old and new, backfill historical data. Phase 3 (Contract) — once all application versions use the new schema, remove the old column/table. Each phase is a separate deployment, ensuring the application is always compatible with the current schema.
  • Backwards-Compatible Migrations: Every schema change must be compatible with both the current and previous application version. Never rename or remove a column in the same deployment that stops using it. Instead: add the new column (deploy #1), update the app to use the new column (deploy #2), remove the old column (deploy #3). This enables safe rollback at any point.
  • Migration Tools: Use versioned migration tools like Flyway, Liquibase, or Alembic that track which migrations have been applied. Migrations run as part of the CI/CD pipeline, typically before the application deployment. Each migration is an immutable, version-controlled script.
  • Testing: Run migrations against a production-cloned database in the CI pipeline to catch issues before production. Test both the forward migration and the rollback migration. For large tables, test migration performance to ensure it completes within maintenance windows or use online schema change tools (pt-online-schema-change, gh-ost) for zero-downtime migrations.

Q4: How would you design a CI/CD pipeline for a microservices architecture with 50+ services?

Scaling CI/CD across many microservices requires standardization, independence, and shared infrastructure:

  • Independent Pipelines: Each microservice should have its own CI/CD pipeline triggered by changes to its code directory (monorepo) or repository (polyrepo). Services should be independently deployable — changing Service A should not require rebuilding or redeploying Service B. This is enabled by maintaining stable API contracts between services.
  • Standardized Pipeline Templates: Create reusable pipeline templates (GitHub Actions reusable workflows, Jenkins shared libraries) that encapsulate your organization's build, test, security scan, and deployment stages. Each service configures the template with service-specific parameters (language, test command, deployment target) rather than defining the entire pipeline from scratch.
  • Contract Testing: With 50+ services, integration testing every combination is impractical. Use contract testing (Pact, Spring Cloud Contract) where each service independently verifies that it fulfills its provider contracts and that its consumer expectations are met. This replaces expensive end-to-end tests with fast, focused contract verification.
  • Progressive Delivery: Use an automated progressive delivery tool (Argo Rollouts, Flagger) that automates canary analysis. After deployment, the tool monitors SLIs (error rate, latency) and automatically promotes or rolls back the canary based on predefined success criteria — no human monitoring required for each of the 50+ service deployments happening daily.
