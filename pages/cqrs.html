CQRS (Command Query Responsibility Segregation)

Separate read and write models to independently optimize queries and commands for scalability and performance

Definition

What is CQRS?

Command Query Responsibility Segregation (CQRS) is an architectural pattern that separates the read (query) and write (command) operations of a system into distinct models. Instead of using a single data model for both reading and writing data, CQRS uses one model optimized for updates (the write model) and a different model optimized for queries (the read model). Commands change state but return no data. Queries return data but do not change state. This separation allows each side to be independently scaled, optimized, and evolved without impacting the other.

Key Insight: Think of a library system: the cataloging department (write side) organizes and updates book records with detailed metadata, while the public search terminal (read side) presents a simplified, fast-searchable view. Each is optimized for its purpose rather than forcing one system to do both poorly.

Simple Explanation

Understanding the Concept

In traditional applications, the same data model and database handle both reads and writes. This works fine for simple applications, but in complex, high-traffic systems, reads and writes often have very different requirements. Reads may need denormalized data across multiple entities for fast queries, while writes need normalized data with strong validation and business rules. CQRS solves this tension by splitting them apart.

Here is how CQRS works in practice:

  • Command Side (Write Model): Handles all create, update, and delete operations. The write model is typically normalized, enforces business rules and invariants, and is optimized for transactional integrity. Commands are processed through command handlers that validate and execute business logic.
  • Query Side (Read Model): Handles all data retrieval operations. The read model is typically denormalized, pre-computed, and optimized for specific query patterns. It may use different storage technologies like Elasticsearch for full-text search or Redis for fast key-value lookups.
  • Synchronization: When data changes on the write side, the read model must be updated. This is often done asynchronously through domain events. The write side publishes events, and the read side subscribes to them and updates its projections accordingly.
  • Different Data Stores: The read and write sides can use completely different databases. For example, the write side might use PostgreSQL for ACID transactions, while the read side uses MongoDB for flexible document queries or Elasticsearch for search.
  • Event Sourcing Compatibility: CQRS pairs naturally with Event Sourcing, where the write side stores events and the read side builds materialized views (projections) from those events.

Benefits of CQRS:

  • Independent Scaling: Scale read and write sides separately based on their individual load
  • Optimized Models: Each model is tailored for its specific use case without compromise
  • Simplified Queries: Read models can be pre-computed and denormalized for fast retrieval
  • Better Security: Fine-grained control over which operations users can perform

Real-World Example

High-Traffic E-Commerce Platform

Consider an e-commerce platform like Amazon during Black Friday. The product catalog is read millions of times per second (browsing, searching, filtering), while orders are written thousands of times per second. With CQRS, the read side uses Elasticsearch for lightning-fast product searches with denormalized data (product name, price, ratings, images all in one document), while the write side uses a relational database with normalized tables for order processing, inventory management, and payment validation. The read model is updated asynchronously when inventory changes, meaning there's a brief delay before a stock change appears in search results, but the system can handle massive read traffic without impacting order processing performance.

Key Points & Summary

Read/Write Separation

Commands (writes) and queries (reads) use separate models, potentially with different data stores and schemas. Commands enforce business rules and return no data; queries are optimized for retrieval and do not modify state. This clear separation simplifies each model.

Event Sourcing Compatibility

CQRS naturally pairs with Event Sourcing where the write side appends events to an event store and the read side builds materialized views (projections) from those events. Together they provide a complete audit trail and temporal query capability.

Eventual Consistency

Because the read model is updated asynchronously from the write model, there is a propagation delay. The system is eventually consistent: after a write, the read model will reflect the change, but not instantaneously. Design your UI to handle this stale-read window.

When NOT to Use CQRS

Avoid CQRS for simple CRUD applications with balanced read/write loads, systems requiring strong consistency, small teams without distributed systems experience, or domains where the read and write models would be nearly identical.

Quick Summary

  • CQRS separates read and write operations into distinct models and potentially different data stores
  • The write model enforces business rules; the read model is optimized for query performance
  • Read and write sides can be scaled independently based on their traffic patterns
  • Pairs naturally with Event Sourcing for complete audit trails and temporal queries
  • Introduces eventual consistency and added complexity; not suitable for simple CRUD apps

Interview Questions & Answers

What is CQRS and what problem does it solve?

CQRS (Command Query Responsibility Segregation) separates the read and write sides of an application into distinct models. It solves several problems:

  • Mismatched optimization: Read and write operations have fundamentally different performance requirements that a single model cannot satisfy optimally
  • Scaling imbalance: Most applications have vastly more reads than writes (often 100:1 or higher), but a single model forces both to scale together
  • Complex queries: When reads require joining many tables or aggregating data, the queries become slow and complex. A denormalized read model eliminates expensive joins
  • Contention: Read and write operations competing for the same database resources cause lock contention and degraded performance under load

How does CQRS relate to Event Sourcing? Do you need both together?

CQRS and Event Sourcing are complementary but independent patterns:

  • CQRS without Event Sourcing: You can implement CQRS with traditional databases on both sides. The write side uses a normalized relational database, and the read side uses denormalized views or a separate read-optimized store, synchronized via change data capture or domain events.
  • Event Sourcing without CQRS: You can store events as the source of truth without separating read and write models, though this is uncommon since querying an event store directly is typically impractical.
  • Together: They work exceptionally well together. Event Sourcing provides the write model (append-only event store), and CQRS provides separate read models (projections) built by replaying events. Events serve as the natural synchronization mechanism between the two sides.

In practice, Event Sourcing almost always uses CQRS because querying an event stream directly is inefficient for most use cases.

How do you handle eventual consistency in CQRS systems?

Eventual consistency is a fundamental trade-off in CQRS. Strategies to manage it include:

  • UI Optimistic Updates: After a command succeeds, update the UI immediately with the expected state without waiting for the read model to sync. This gives the user instant feedback.
  • Polling or WebSockets: The client polls the read model or subscribes to real-time updates until the read model reflects the latest write.
  • Read-Your-Writes: After a command, include a version token. The query side checks if the read model has caught up to that version before returning data.
  • Synchronous Read Model Update: For critical paths, update the read model synchronously within the command handler, sacrificing some decoupling for immediate consistency.
  • Business Process Design: Design workflows that naturally accommodate delays, such as showing "Order Processing" status instead of immediate confirmation.

When should you NOT use CQRS?

CQRS adds significant complexity and should be avoided in these scenarios:

  • Simple CRUD applications: If your app is basic create-read-update-delete with no complex business logic, CQRS is overkill
  • Balanced read/write loads: If reads and writes are roughly equal and don't have conflicting optimization needs, separation provides little benefit
  • Strong consistency requirements: If every read must immediately reflect the latest write (e.g., financial account balances), eventual consistency is problematic
  • Small teams: CQRS requires expertise in distributed systems, event-driven architecture, and operational tooling that small teams may lack
  • Early-stage products: When the domain model is still evolving rapidly, maintaining two separate models doubles the refactoring effort
